Namespaces & Declaration Files

Before ES6 modules, TypeScript used namespaces (namespace keyword) to organize code into logical groups. Today, ES6 modules (import/export) have largely replaced namespaces for application code. Namespaces still appear in legacy codebases and .d.ts files for global library typings.

Declaration files (.d.ts) describe the shape of JavaScript libraries without any implementation — they contain only type information. The DefinitelyTyped repository (@types/* packages) is the primary source of community-maintained declaration files.

1 What are namespaces in TypeScript and are they still used?

Namespaces are TypeScript's original way to organize code into logical groups using the namespace keyword. They prevent global scope pollution by wrapping related types, interfaces, and functions into a named container. You access members via dot notation (MyNamespace.myFunction()).

In modern code, ES6 modules have largely replaced namespaces. Use import/export for new projects. Namespaces are still useful for organizing .d.ts files and in legacy codebases. 

// Legacy pattern — namespaces
namespace Validation {
  export interface StringValidator {
    isValid(s: string): boolean;
  }
  export function validateEmail(s: string): boolean {
    return s.includes("@");
  }
}

// Modern pattern — ES6 modules
// validators.ts
export interface StringValidator { ... }
export function validateEmail(s: string): boolean { ... }
💡 Memory Hook: Namespaces = legacy organizational tool. Prefer ES6 modules (import/export) for new code. Think of namespaces as the "old school" way before import/export existed in JS.

2 What are .d.ts declaration files?

.d.ts files are declaration files that describe the type shape of JavaScript code without containing any implementation. They tell TypeScript "this JS library exists and here's its API." When you install @types/lodash, you're installing a .d.ts file that describes Lodash's types.

Key properties: no runtime code (no JS output), only type declarations. TypeScript automatically loads .d.ts files found in your project or node_modules/@types. 

// lodash.d.ts — type descriptions only, no implementation
declare module 'lodash' {
  export function debounce<T extends (...args: any[]) => void>(
    func: T,
    wait?: number
  ): T;
  export function cloneDeep<T>(value: T): T;
}
💡 Memory Hook: .d.ts = type description without code. Think of it as a restaurant menu — it tells you what's available, but isn't the food itself.

3 What does strictNullChecks do?

When strictNullChecks is enabled, null and undefined are no longer assignable to every type. Without it, let name: string = null is legal (dangerous!). With it, you must explicitly opt in: let name: string | null = null.

This eliminates an entire category of bugs — the infamous TypeError: Cannot read property 'x' of null/undefined. The compiler forces you to handle null cases explicitly via narrowing (if (value !== null)), optional chaining (value?.prop), or nullish coalescing (value ?? default). 

// strictNullChecks: false (dangerous)
let name: string = null; // OK — no error!
name.toUpperCase(); // Runtime crash: Cannot read property of null

// strictNullChecks: true (safe)
let name: string = null; // Error! null not assignable to string
let name: string | null = null; // OK — explicit null handling
if (name !== null) name.toUpperCase(); // Safe after narrowing
💡 Memory Hook: strictNullChecks = "null is not a valid string/number/etc." Always enable it. It's the single biggest safety improvement in TypeScript.

4 What are recursive types?

Recursive types are types that reference themselves in their own definition. They model nested, self-similar data structures like trees, linked lists, JSON objects, and file systems. The type contains a property of the same type, creating a structure that can be infinitely deep. 

// Binary tree node
interface TreeNode {
  value: number;
  left?: TreeNode;  // references itself
  right?: TreeNode;
}

// Linked list node
interface ListNode<T> {
  data: T;
  next?: ListNode<T>; // references itself
}

// Usage — tree of depth 3
const tree: TreeNode = {
  value: 1,
  left: { value: 2, left: { value: 4 } },
  right: { value: 3 }
};
💡 Memory Hook: Recursive types = self-referencing types. If the data nests inside itself (trees, lists, JSON), use a recursive type. Like a mirror reflecting a mirror.

5 What are variadic tuple types?

Variadic tuple types allow spreading generic type parameters inside tuple definitions. This lets you write type-level functions that manipulate tuples — concatenating them, prepending/appending elements, or transforming their contents — while preserving full type information. 

// Concat two tuples at the type level
type Concat<A extends readonly unknown[], B extends readonly unknown[]> =
  [...A, ...B];

type Result = Concat<[string, number], [boolean]>;
// Result = [string, number, boolean]

// Prepend a head element
type Prepend<T, Arr extends readonly unknown[]> = [T, ...Arr];
type WithHead = Prepend<string, [number, boolean]>;
// WithHead = [string, number, boolean]

// Practical use: typed function argument piping
function pipe<T extends readonly unknown[]>(
  ...args: [...T, (...params: T) => void]
) { /* ... */ }
💡 Memory Hook: Variadic tuples = generic spread in tuples. Think of them as "type-level array manipulation" — concat, prepend, append while keeping exact types.

6 How do you perform exhaustive checking in TypeScript?

Exhaustive checking uses the never type to ensure all cases of a union are handled in a switch or if-else chain. In the default case, assign the value to a never-typed variable. If you later add a new member to the union and forget to add a case, TypeScript will produce a compile error because the new type can't be assigned to never. 

type Shape = "circle" | "square" | "triangle";

function getArea(shape: Shape): number {
  switch (shape) {
    case "circle":   return Math.PI;
    case "square":   return 1;
    case "triangle": return 0.5;
    default:
      // If a new shape is added to the union but not handled above,
      // this assignment will fail to compile:
      const _exhaustive: never = shape;
      throw new Error(`Unhandled shape: ${_exhaustive}`);
  }
}

// Add "hexagon" to Shape → compile error in default case!
💡 Memory Hook: never in default = compiler-enforced exhaustive check. If you add a new union member and forget a case, the compiler catches it immediately.

7 What are branded types and why do they exist?

TypeScript uses structural typing (duck typing) — if two types have the same shape, they're compatible. Branded types (also called opaque types or nominal types) add a unique marker to a type so that even identical structures become incompatible. This prevents accidentally passing a UserId where an OrderId is expected, even though both are string. 

// Branded type pattern
type Brand<T, B> = T & { __brand: B };

type UserId = Brand<string, "UserId">;
type OrderId = Brand<string, "OrderId">;

function getUser(id: UserId) { /* ... */ }
function getOrder(id: OrderId) { /* ... */ }

const uid = "abc" as UserId;
const oid = "xyz" as OrderId;

getUser(uid); // OK
getUser(oid); // Error! OrderId not assignable to UserId
💡 Memory Hook: Branded types = same structure, different identity. Like a passport — it's still paper with your name, but the country code makes it unique.

8 What are TypeScript access modifiers and how do they work?

TypeScript has four access levels: public (default, accessible everywhere), private (class only, compile-time check only), protected (class + subclasses), and # prefix (ECMAScript private, runtime privacy enforced by the JS engine).

Critical distinction: TypeScript's private keyword is a compile-time illusion — it's erased at runtime, so JavaScript code can still access the property. The # prefix (e.g., #secret) is real runtime privacy enforced by the JavaScript engine itself. 

class Wallet {
  public owner: string;       // default — accessible anywhere
  private balance: number;    // compile-time only — not in .js
  protected id: string;      // class + subclasses only
  #secret: string;            // TRUE privacy — runtime enforced

  constructor(owner: string) {
    this.owner = owner;
    this.balance = 0;
    this.id = crypto.randomUUID();
    this.#secret = "hidden";
  }
}
💡 Memory Hook: TS private = compile-time illusion (erased at runtime). #private = real privacy (enforced by JS engine). Use # when you need actual runtime encapsulation.

9 What typing system does TypeScript use?

TypeScript uses structural typing (duck typing) — type compatibility is determined by the shape of types, not their names or declarations. If type A has all the properties that type B requires, A is assignable to B, regardless of what A or B are called. This is the opposite of nominal typing (used by Java, C#) where types must share a name or inheritance chain. 

interface Named { name: string; }
interface Person { name: string; age: number; }

const p: Person = { name: "Ali", age: 25 };

// Person is assignable to Named — same shape
const n: Named = p; // OK — structural compatibility

// For nominal behavior, use branded types
type USD = Brand<number, "USD">;
type EUR = Brand<number, "EUR">;
// Both are number, but now incompatible!
💡 Memory Hook: TypeScript = "if it looks like a duck, it is a duck". Shape matters, not name. Want name-based safety? Use branded types.

10 What does emitDecoratorMetadata do?

emitDecoratorMetadata is a TypeScript compiler option that embeds type information into the emitted JavaScript as decorator metadata. When enabled, the compiler adds design:type, design:paramtypes, and design:returntype metadata entries that decorators can read at runtime. This is heavily used by frameworks like NestJS and TypeORM for dependency injection and ORM mapping. 

// tsconfig.json
{
  "compilerOptions": {
    "experimentalDecorators": true,
    "emitDecoratorMetadata": true
  }
}

// NestJS uses this for automatic DI
@Injectable()
class UserService {
  constructor(private repo: UserRepository) {}
  //                  ^ type metadata emitted → NestJS resolves
  //                    the dependency automatically at runtime
}
💡 Memory Hook: emitDecoratorMetadata = runtime type info for decorators. Enables frameworks (NestJS, TypeORM) to "see" TypeScript types at runtime for DI and ORM mapping.

11 What is type-level programming in TypeScript?

Type-level programming means using TypeScript's type system as a mini programming language — writing logic that "runs" entirely in the compiler, not at runtime. You use conditional types, mapped types, recursive types, template literal types, and inference to compute new types from existing ones. The output is a type, not a value. 

// Type-level string manipulation
type Capitalize<S extends string> =
  S extends `${infer First}${infer Rest}`
    ? `${Uppercase<First>}${Rest}`
    : S;

type Result = Capitalize<"hello">; // "Hello"

// Type-level arithmetic (depth limit)
type Length<T extends any[]> = T['length'];
type Three = Length<[1, 2, 3]>; // 3 (number literal type)
💡 Memory Hook: Type-level programming = "code that runs in the compiler". Input = types, output = types. No JS is generated — it's purely compile-time computation.

12 What is module augmentation?

Module augmentation lets you add types to someone else's module — extending or modifying the type declarations of third-party libraries (or your own modules) without editing the original source. It uses declare module 'library-name' to add new properties, functions, or interfaces to an existing module's type definitions. 

// Extend Express Request with custom properties
declare module 'express' {
  interface Request {
    user?: {
      id: string;
      role: 'admin' | 'user';
    };
    sessionId?: string;
  }
}

// Now req.user and req.sessionId are typed everywhere
app.get('/profile', (req, res) => {
  const userId = req.user?.id; // Fully typed!
});
💡 Memory Hook: Module augmentation = "add types to someone else's module". Like adding a sticky note to a library book — you extend it without modifying the original.

🧪 Quick Check

1. What does strictNullChecks prevent?

✅ Correct! strictNullChecks makes null and undefined their own distinct types — they can't silently sneak into string, number, etc.
❌ strictNullChecks prevents null and undefined from being assignable to other types. You must explicitly opt in with T | null.

2. What is the purpose of assigning to never in a switch default?

✅ Correct! Assigning to never in the default case causes a compile error whenever a new union member is added but not handled.
❌ The never assignment in default is an exhaustive check — if a new union member is added but not handled in the switch, TypeScript produces a compile error.

3. What is the key difference between TypeScript's private and # private fields?

✅ Correct! TypeScript's private is erased at compile time — no runtime protection. # fields are ECMAScript standard and enforced by the JS engine itself.
❌ TypeScript's private is a compile-time check only (erased in JS output). The # prefix provides real runtime privacy enforced by the JavaScript engine.

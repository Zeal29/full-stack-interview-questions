Defining Types

Interfaces, types, assertions, unions, intersections, and enums

What You'll Learn

  • Choose between interface and type for different scenarios
  • Use type assertion safely and understand its limitations
  • Create union and intersection types and know when to use each
  • Use enums and understand the preference for const enums / union types

Why Define Custom Types?

TypeScript's built-in types (string, number, boolean) cover primitives, but real applications work with complex shapes — objects with specific properties, values from a fixed set, or combinations of multiple types.

TypeScript gives you four main tools to define custom types: interfaces, type aliases, enums, and compositions (unions & intersections). Choosing the right tool for each job is a core TypeScript skill.

The guiding principle: start with what you need now, compose for what you need later. A simple interface or type alias beats a clever generic 95% of the time.

1 What is the difference between interface and type?

Both define object shapes, but they differ in capabilities:

Interface: Designed for object shapes. Extensible via extends, supports declaration merging (multiple declarations with the same name combine automatically). Best for public APIs and class contracts.

Type: More flexible — supports unions (|), intersections (&), mapped types, conditional types, and tuple definitions. Cannot be re-opened after creation. Best for complex compositions. 

// Interface — extensible, supports merging
interface Animal { name: string }
interface Animal { age: number } // merges with above!
interface Dog extends Animal { breed: string }

// Type — supports unions, intersections, mapped types
type Status = "active" | "inactive";
type Employee = { name: string } & { age: number };
💡 Memory Hook: Interface = "I describe an object" (open, extensible). Type = "I can be anything" (flexible, compositional). Use interface for public APIs, type for complex compositions.

2 What is type assertion and when should you use it?

Type assertion (as Type) tells TypeScript: "trust me, I know the type better than you do." It is a compiler hint only — it does not change the runtime type or perform any conversion. No casting happens at runtime.

Use it sparingly — typically when you have more information than the compiler (e.g., DOM elements, parsed JSON from a known API shape). Overusing it defeats type safety. 

// DOM example — compiler only knows it's HTMLElement
const input = document.getElementById("email") as HTMLInputElement;
input.value; // OK — TS now knows it has .value

// ⚠️ This does NOT convert the value!
const x = "hello" as number; // Error — unrelated types
💡 Memory Hook: as = "I know better than you" to TypeScript. It's a compiler-only hint — it never changes the runtime type. Use sparingly.

3 What are union types and how do you use them?

A union type (A | B) allows a value to be one of several types. You must narrow the type before using type-specific operations. Type guards include typeof, in, and instanceof. 

function format(value: string | number): string {
  if (typeof value === "string") {
    return value.trim(); // TS knows it's string here
  }
  return value.toFixed(2); // TS knows it's number here
}

// Literal union — restricted set of strings
type Direction = "left" | "right" | "up" | "down";
💡 Memory Hook: Union = OR (|). The value can be this or that. Think of it as a menu — pick one.

4 What are intersection types and how do they differ from unions?

An intersection type (A & B) combines ALL properties from every member type. The resulting value must satisfy every type in the intersection. Use it to compose types together — the value has everything. 

type Named = { name: string };
type Aged = { age: number };
type Employee = Named & Aged;

const emp: Employee = {
  name: "Ali",  // from Named
  age: 25       // from Aged
};

Key difference: Union = pick one (|). Intersection = have all (&).

💡 Memory Hook: Intersection = AND (&). The value must have this and that. Think of it as a checklist — must have everything.

5 What are enums and when should you prefer alternatives?

Enums define a set of named constants. TypeScript supports numeric enums (auto-incrementing), string enums, and const enums (inlined at compile time). 

// Numeric enum — auto-incrementing
enum Direction { Up, Down, Left, Right }
// Up=0, Down=1, Left=2, Right=3

// String enum — explicit values
enum Status { Active = "ACTIVE", Inactive = "INACTIVE" }

// Const enum — inlined, no JS output
const enum Color { Red, Green, Blue }
let c = Color.Red; // compiles to: let c = 0;

Best practice: Prefer union types (type Status = "active" | "inactive") or const enums over regular enums. Regular enums generate extra JavaScript and can hinder tree-shaking.

💡 Memory Hook: Prefer union types or const enums — they tree-shake cleanly and add zero runtime overhead.

🧪 Quick Check

1. Which feature does type support that interface does not?

✅ Correct! type supports unions, intersections, mapped types, and conditional types — none of which interface can express directly.
❌ type uniquely supports unions (|), intersections (&), mapped types, and conditional types. Both interface and type can describe objects and be used as parameter types.

2. What does type assertion (as) actually do at runtime?

✅ Correct! Type assertion is erased at compile time — it tells TypeScript to trust you, but changes nothing at runtime.
❌ Type assertion (as) does nothing at runtime — it's purely a compile-time hint. No conversion, no checking, no error.

3. Which is the best practice for defining a fixed set of string values?

✅ Correct! A string literal union is the cleanest approach — zero runtime overhead, full type safety, and tree-shakes perfectly.
❌ Union types of string literals ("active" | "inactive") are preferred over regular enums — they generate no JavaScript and integrate cleanly with tooling.

🟢 Easy Puzzles

Puzzle 1 — What's the Type? easy

Look at this code. What is the type of result? Not the runtime value — the TypeScript type. 

type A = { name: string; age: number };
type B = { name: string; email: string };

type Result = A & B;

// What is the type of Result?

Result is an intersection type that combines both A and B. 

// Result is equivalent to:
{
  name: string;   // from both A and B (same type — compatible)
  age: number;   // from A
  email: string;  // from B
}

Why: The & operator creates an intersection — a value must satisfy both types. The name property appears in both but with the same type (string), so it merges cleanly. If the types conflicted (e.g., name: string vs name: number), the resulting property would be never — no value can be both.

💡 Key Insight: Intersection (&) = AND. Union (|) = OR. A & B requires all properties from both. A | B requires properties from either.
Puzzle 2 — Fix the Type Error easy

This code has a type error. What's wrong, and how do you fix it with minimal changes? 

interface User {
  id: number;
  name: string;
  role: "admin" | "user";
}

const user: User = {
  id: 1,
  name: "Alice",
};

The role property is required in the interface but missing from the object literal. TypeScript error: "Property 'role' is missing in type... but required in type 'User'".

Fix — add the missing property:

const user: User = {
  id: 1,
  name: "Alice",
  role: "admin",   // ✅ must be "admin" or "user"
};

Alternative fixes (not minimal, but valid): Make role optional with role?: "admin" | "user", or give it a default. But the minimal fix is just adding the missing property.

💡 Key Insight: TypeScript enforces structural typing strictly — every required property must be present. Optional properties (marked with ?) can be omitted.
Puzzle 3 — What Type Satisfies This? easy

Write a type Status that meets all these requirements:

  • Must be one of three string values: "pending", "active", "closed"
  • Must NOT accept arbitrary strings like "unknown"
  • Must be usable as an object key
// Your type here:
type Status = /* ??? */;

// These should work:
const s1: Status = "pending";
const s2: Status = "active";

// This must fail:
const s3: Status = "unknown"; // ❌ Error

Answer:

type Status = "pending" | "active" | "closed";

Why this works: This is a string literal union. It's narrower than string — only those three exact values are assignable. Any other string produces a compile error. String literals are valid object keys because they're subtypes of string.

Why not enum? Enums work too, but string literal unions are lighter — no runtime cost, no generated JS code, and they're the idiomatic modern TypeScript approach.

💡 Key Insight: String literal unions are TypeScript's way of creating closed sets of values. They give you enum-like safety without enum's runtime overhead.
Puzzle 4 — Predict the Error easy

Which line has a type error, and why? 

function greet(name: string | null): string {
  return "Hello, " + name.toUpperCase();  // Line A
}

const x: string | number = 42;
const y: string = x;                    // Line B

Both lines error.

Line A: name is string | null. Calling .toUpperCase() on null crashes at runtime, so TypeScript rejects it. Fix: narrow with if (name !== null) or use the optional chain name?.toUpperCase() ?? "stranger".

Line B: x is typed as string | number. Assigning it to y: string is unsafe — what if x is 42 (a number)? TypeScript won't allow this without explicit narrowing. 

// Fixed Line A:
function greet(name: string | null): string {
  if (name === null) return "Hello, stranger";
  return "Hello, " + name.toUpperCase(); // ✅ narrowed to string
}

// Fixed Line B:
if (typeof x === "string") {
  const y: string = x; // ✅ narrowed to string
}
💡 Key Insight: Union types require narrowing before accessing members. TypeScript forces you to prove which member of the union you're working with.

Puzzle 5 — Write a Generic Safe Getter medium

Write a generic function getNestedValue that safely retrieves a nested property from an object. It should return undefined if any part of the path doesn't exist. No any allowed. 

// Implement this:
function getNestedValue<T, K1 extends keyof T>(
  obj: T,
  key: K1
): /* ??? */;

// Usage:
const data = { user: { profile: { name: "Alice" } } };
getNestedValue(data, "user");            // { profile: { name: "Alice" } }
getNestedValue(data.user.profile, "name"); // "Alice"

Single-level safe getter:

function getNestedValue<T, K extends keyof T>(
  obj: T,
  key: K
): T[K] | undefined {
  return obj?.[key];
}

For deep nesting, a more practical approach uses a dot-path string with recursive conditional types: 

type DeepValue<T, P extends string> =
  P extends "" ? T :
  P extends "" ? T :
  P extends infer Key extends infer K ?
    K extends keyof T ?
      P extends " " ?
        T[K] :
        T[K]
    : undefined
  : never;

// Simpler practical approach with type assertion:
function get<T>(obj: unknown, path: string): T | undefined {
  const keys = path.split(".");
  let result: unknown = obj;
  for (const key of keys) {
    if (result == null || typeof result !== "object") {
      return undefined;
    }
    result = (result as Record<string, unknown>)[key];
  }
  return result as T | undefined;
}

Why: The generic version ensures key is a valid key of T — passing a non-existent key is a compile error. The return type T[K] preserves the exact type of the property. Using unknown instead of any for intermediate values keeps type safety.

💡 Key Insight: keyof T creates a union of all keys of T. Using K extends keyof T constrains the key parameter to only valid property names.
Puzzle 6 — Design a Discriminated Union Event System medium

Design a type-safe event system using discriminated unions. Each event type should have a type field and a different payload shape. Then write a handleEvent function that TypeScript can fully narrow. 

// Define these types:
type UserCreatedEvent = /* ??? */;
type UserDeletedEvent = /* ??? */;
type UserUpdatedEvent = /* ??? */;
type AppEvent = /* ??? */;

// This function must be fully type-safe (no assertions):
function handleEvent(event: AppEvent): void {
  // implement
}
type UserCreatedEvent = {
  type: "user_created";
  payload: { id: string; name: string; email: string };
};

type UserDeletedEvent = {
  type: "user_deleted";
  payload: { id: string; reason: string };
};

type UserUpdatedEvent = {
  type: "user_updated";
  payload: { id: string; changes: Record<string, unknown> };
};

type AppEvent = UserCreatedEvent | UserDeletedEvent | UserUpdatedEvent;

function handleEvent(event: AppEvent): void {
  switch (event.type) {
    case "user_created":
      console.log("Created:", event.payload.name);   // ✅ payload.name exists
      break;
    case "user_deleted":
      console.log("Deleted:", event.payload.reason); // ✅ payload.reason exists
      break;
    case "user_updated":
      console.log("Updated:", event.payload.changes); // ✅ payload.changes exists
      break;
  }
}

Why this works: The type field is the discriminant — a common property with a unique literal value on each variant. When you check event.type in a switch, TypeScript narrows the union to that specific variant, so event.payload knows its exact shape. No type assertions needed.

💡 Key Insight: Discriminated unions = a shared literal field + switch/if narrowing. This pattern is TypeScript's answer to sum types / algebraic data types. It's how Redux actions, AST nodes, and network responses are typed in real codebases.
Puzzle 7 — Fix the Type Narrowing medium

This function is supposed to safely format user input, but TypeScript errors on every property access. Add the minimal type guards to make it compile without errors — no any, no type assertions. 

function formatInput(input: unknown): string {
  // ERROR: Property 'name' does not exist on type 'unknown'
  if (input.name !== undefined) {
    return input.name.toUpperCase();
  }
  // ERROR: Property 'length' does not exist on type 'unknown'
  if (input.length > 0) {
    return String(input);
  }
  return "default";
}
function formatInput(input: unknown): string {
  // First: verify input is an object (not null, not primitive)
  if (
    typeof input === "object" &&
    input !== null &&
    "name" in input
  ) {
    const obj = input as { name: unknown };
    if (typeof obj.name === "string") {
      return obj.name.toUpperCase(); // ✅
    }
  }

  // For length check: verify it's a string or array
  if (typeof input === "string" && input.length > 0) {
    return input; // ✅ already a string
  }

  return "default";
}

Why the original fails: unknown doesn't allow any property access. You must narrow first. The typeof check narrows primitives. The in operator narrows objects. Each check peels away uncertainty.

The narrowing chain: unknown → typeof === "object" → !== null → "name" in input → now TypeScript knows it's an object with a name property.

💡 Key Insight: typeof null === "object" is a JS quirk. Always check input !== null after typeof === "object". The in operator is the cleanest way to check for property existence in type guards.
Puzzle 8 — Utility Type Composition medium

You have a full User interface. Create a update user type by composing utility types. Requirements:

  • Start from User below
  • The id field must be required
  • All other fields must be optional
  • The email field should be excluded entirely (it's updated via a separate flow)
interface User {
  id: number;
  name: string;
  email: string;
  role: "admin" | "editor" | "viewer";
  isActive: boolean;
}

// Create this type using Pick, Omit, and Partial:
type UpdateUserPayload = /* ??? */;
type UpdateUserPayload =
  & { id: number }
  & Partial<Omit<User, "id" | "email">>;

// Equivalent to:
// {
//   id: number;          // required
//   name?: string;       // optional
//   role?: "admin" | "editor" | "viewer";  // optional
//   isActive?: boolean;  // optional
//   // no email!
// }

How it builds up:

  1. Omit<User, "id" | "email"> — removes both id and email, leaving { name, role, isActive }
  2. Partial<...> — makes all remaining fields optional
  3. { id: number } & ... — adds id back as required via intersection

Why composition over a hand-written type: If User gains new fields, UpdateUserPayload automatically picks them up as optional. Single source of truth.

💡 Key Insight: Utility types compose like functions: Partial(Omit(User, X)) is "remove X, then make optional". Think left-to-right, inner-to-outer.

Puzzle 9 — Write DeepReadonly<T> hard

TypeScript's built-in Readonly<T> only makes the top level readonly. Write DeepReadonly<T> that makes all nested properties readonly — recursively. 

// Built-in Readonly only does one level:
type Shallow = Readonly<{ nested: { value: string } }>;
// { readonly nested: { value: string } }
// nested itself is NOT readonly — you can mutate nested.value

// Your DeepReadonly should produce:
// { readonly nested: { readonly value: string } }

type DeepReadonly<T> = /* ??? */;

// Test it:
type Config = {
  db: {
    host: string;
    port: number;
    credentials: {
      user: string;
      pass: string;
    };
  };
  debug: boolean;
};

type FrozenConfig = DeepReadonly<Config>;
// All properties at every depth should be readonly
type DeepReadonly<T> = T extends Function
  ? T
  : T extends object
    ? { readonly [K in keyof T]: DeepReadonly<T[K]> }
    : T;

How it works (line by line):

  1. T extends Function ? T — pass functions through unchanged (don't try to make function properties readonly)
  2. T extends object ? { readonly [K in keyof T]: DeepReadonly<T[K]> } — if it's an object, map every property to readonly and recurse on the value type
  3. : T — primitives (string, number, boolean, etc.) pass through unchanged

Why the Function check matters: Without it, methods become { readonly bind: DeepReadonly<...> } which breaks them. Functions are objects in JS, so the extends object check catches them too. 

// Result for FrozenConfig:
{
  readonly db: {
    readonly host: string;
    readonly port: number;
    readonly credentials: {
      readonly user: string;
      readonly pass: string;
    };
  };
  readonly debug: boolean;
}
💡 Key Insight: Recursive conditional types follow a pattern: base case (primitive) → recursive case (object) → special case (Function/Array). Always handle edge cases that are technically objects but shouldn't be recursed into.
Puzzle 10 — Type-Safe Event Emitter hard

Build a type-safe event emitter using generics and discriminated unions. The API must ensure:

  • on(event, handler) — handler receives the correct payload type for that event
  • emit(event, payload) — payload must match the event's expected type
  • Wrong event name = compile error
  • Wrong payload shape = compile error
// The event map that drives everything:
interface EventMap {
  login: { userId: string; timestamp: number };
  logout: { userId: string };
  error: { message: string; code: number };
}

// Implement the Emitter class:
class Emitter<T> { /* ??? */ }

// Usage must be fully type-safe:
const emitter = new Emitter<EventMap>();
emitter.on("login", (payload) => {
  console.log(payload.userId); // ✅ typed as string
});
emitter.emit("error", { message: "fail", code: 500 }); // ✅
emitter.emit("login", { message: "fail" }); // ❌ wrong payload
type Handler<T> = (payload: T) => void;

class Emitter<TEventMap extends Record<string, unknown>> {
  private listeners = new Map<
    keyof TEventMap,
    Set<Handler<unknown>>
  >();

  on<K extends keyof TEventMap>(
    event: K,
    handler: Handler<TEventMap[K]>
  ): void {
    if (!this.listeners.has(event)) {
      this.listeners.set(event, new Set());
    }
    this.listeners.get(event)!.add(
      handler as Handler<unknown>
    );
  }

  emit<K extends keyof TEventMap>(
    event: K,
    payload: TEventMap[K]
  ): void {
    this.listeners.get(event)?.forEach(
      (handler) => handler(payload)
    );
  }
}

How the type safety works:

  1. TEventMap extends Record<string, unknown> constrains the map to string-keyed objects
  2. K extends keyof TEventMap ensures only valid event names
  3. TEventMap[K] in the handler/emit signature ensures the payload type matches the event
  4. When you call on("login", handler), TypeScript infers K = "login" and resolves TEventMap["login"] to { userId: string; timestamp: number }

The runtime Handler<unknown> cast in on is necessary because Map stores heterogeneous handlers — each event has a different payload type. This is a rare case where a type assertion at the storage boundary is acceptable because the type safety is enforced at the API boundary.

💡 Key Insight: The pattern method<K extends keyof Map>(key: K, value: Map[K]) is the keyed generic pattern — it links the key to the value type. Use it whenever you have a map/registry and want per-key type safety.
Puzzle 11 — Branded Type System hard

Implement branded types (also called opaque types) to prevent mixing up IDs. A UserId should not be assignable to OrderId, even though both are strings at runtime. 

// Define Brand, UserId, and OrderId:
type Brand<T, B> = /* ??? */;

type UserId = /* ??? */;
type OrderId = /* ??? */;

// Helper to create branded values:
function asUserId(id: string): UserId { /* ??? */ }
function asOrderId(id: string): OrderId { /* ??? */ }

// These must work:
const uid: UserId = asUserId("u-123");
const oid: OrderId = asOrderId("o-456");

// This must be a type error:
function getUser(id: UserId): void {}
getUser(oid); // ❌ should error — OrderId is not UserId
// Brand: intersection with a phantom property that has a unique symbol
type Brand<T, B extends string> = T & { readonly __brand: B };

type UserId = Brand<string, "UserId">;
type OrderId = Brand<string, "OrderId">;

// Smart constructors — the ONLY way to create branded values
function asUserId(id: string): UserId {
  return id as UserId;
}

function asOrderId(id: string): OrderId {
  return id as OrderId;
}

// Usage:
const uid: UserId = asUserId("u-123");
const oid: OrderId = asOrderId("o-456");

function getUser(id: UserId): void { /* ... */ }
getUser(oid); // ❌ Error: Type '"OrderId"' is not assignable to type '"UserId"'

How it works: Brand<T, B> creates an intersection of the base type T with a phantom object containing __brand: B. At runtime, this phantom property doesn't exist (strings don't have __brand). But at the type level, UserId and OrderId are different because their __brand literal types differ.

The assertion in constructors (id as UserId) is the controlled escape hatch. You centralize it in one place — the smart constructor — and everywhere else in the codebase gets full type safety. This is a valid use of assertions because the boundary is small and auditable.

💡 Key Insight: Branded types are how production codebases prevent primitive obsession — mixing up values that share the same runtime type but represent different domains. Libraries like io-ts and zod use this pattern for runtime + type safety.
Puzzle 12 — Type-Safe API Client with Template Literals hard

Build a type-safe API client that uses template literal types and conditional types to map API routes to their method signatures. The routes follow a convention:

  • "GET /users" → returns User[], no body
  • "GET /users/:id" → returns User, requires { id: string } param
  • "POST /users" → returns User, requires CreateUserDTO body
  • "DELETE /users/:id" → returns void, requires { id: string } param
interface Routes {
  "GET /users": { response: User[] };
  "GET /users/:id": { response: User; params: { id: string } };
  "POST /users": { response: User; body: CreateUserDTO };
  "DELETE /users/:id": { response: void; params: { id: string } };
}

// Implement a type that extracts the method signature:
type ApiMethod<R extends keyof Routes> = /* ??? */;

// Implement the client type:
type ApiClient = /* ??? */;

// Usage must be fully type-safe:
const client: ApiClient = /* ... */;
client.getUsers();                         // ✅ returns User[]
client.getUsersById({ id: "1" });          // ✅ returns User
client.postUsers({ body: { name: "A" } }); // ✅ returns User
interface User { id: string; name: string; }
interface CreateUserDTO { name: string; }

interface Routes {
  "GET /users": { response: User[] };
  "GET /users/:id": { response: User; params: { id: string } };
  "POST /users": { response: User; body: CreateUserDTO };
  "DELETE /users/:id": { response: void; params: { id: string } };
}

// Step 1: Convert route string to camelCase method name
type RouteToMethod<S extends string> =
  S extends "" ? "" :
  S extends " " ? "" :
  S extends infer First extends string ?
    First extends "/" | ":" ? First :
    First
  : never;

// Simpler approach: use a mapped type with manual method names
type ApiMethod<R extends keyof Routes> =
  "params" extends keyof Routes[R] ?
    "body" extends keyof Routes[R] ?
      (args: { params: Routes[R]["params"]; body: Routes[R]["body"] }) => Promise<Routes[R]["response"]> :
      (args: { params: Routes[R]["params"] }) => Promise<Routes[R]["response"]> :
    "body" extends keyof Routes[R] ?
      (args: { body: Routes[R]["body"] }) => Promise<Routes[R]["response"]> :
      () => Promise<Routes[R]["response"]>;

// Step 2: Full client using template literals to generate method names
type ParseRoute<Route extends string> =
  Route extends "" ? ""
  : Route extends "GET /users" ? "getUsers"
  : Route extends "GET /users/:id" ? "getUsersById"
  : Route extends "POST /users" ? "postUsers"
  : Route extends "DELETE /users/:id" ? "deleteUsersById"
  : never;

// Step 3: Build the client object type
type ApiClient = {
  [R in keyof Routes as ParseRoute<R>]: ApiMethod<R>;
};

// The resulting ApiClient type is:
// {
//   getUsers: () => Promise<User[]>;
//   getUsersById: (args: { params: { id: string } }) => Promise<User>;
//   postUsers: (args: { body: CreateUserDTO }) => Promise<User>;
//   deleteUsersById: (args: { params: { id: string } }) => Promise<void>;
// }

How it works:

  1. ApiMethod<R> uses conditional types to check whether the route config has params, body, both, or neither, and builds the correct function signature
  2. ParseRoute<Route> maps the route string to a camelCase method name using template literal matching
  3. ApiClient uses key remapping (as ParseRoute<R>) to transform the route keys into method names while preserving the type-safe signatures

The key remapping syntax { [R in keyof Routes as ParseRoute<R>]: ApiMethod<R> } is what makes this work — it iterates over each route, transforms the key name, and generates the correct method type.

💡 Key Insight: Key remapping (as clause in mapped types, TS 4.1+) + template literal types let you transform string keys at the type level. This is how frameworks like tRPC and Next.js App Router achieve end-to-end type safety.

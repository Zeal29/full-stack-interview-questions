TypeScript Patterns Overview

TypeScript offers several advanced patterns that go beyond basic type annotations. These patterns let you extend existing types without modifying source files, validate types while preserving specificity, add metadata to classes via decorators, and create precise function signatures. Mastering these patterns is essential for working with large codebases and popular frameworks.

This lesson covers six key patterns: declaration merging, the typeof type operator, the satisfies operator, abstract classes vs interfaces, decorators, and function overloading.

1 What is declaration merging in TypeScript?

Declaration merging means that when you declare two interfaces with the same name, TypeScript automatically combines their members into a single interface. This does not work with type aliases — only interfaces merge.

This is commonly used to extend third-party library types without modifying the original source files. For example, augmenting a module's exported interface in your own project. 

// Original interface (e.g., in a library)
interface Window {
  title: string;
}

// Your extension — merged automatically
interface Window {
  customProp: number;
}

// Now Window has both 'title' and 'customProp'
const w: Window = { title: "app", customProp: 42 };
💡 Memory Hook: Only interfaces merge, not types. Think of it as "interfaces are open for extension, types are sealed."

2 How does typeof work in TypeScript type positions?

In TypeScript's type position, typeof extracts the type from a runtime value. It creates a type that mirrors the shape of the value. This is different from JavaScript's typeof, which returns a string like "string" or "object" at runtime.

This pattern is especially useful for deriving types from configuration objects or constants — write the value once, infer the type automatically. 

const config = {
  host: "localhost",
  port: 3000,
  debug: true,
};

// typeof extracts the shape as a type
type Config = typeof config;
// { host: string; port: number; debug: boolean }

const myConfig: Config = {
  host: "prod.example.com",
  port: 8080,
  debug: false,
};
💡 Memory Hook: typeof in type position = "copy the type of this value". JS typeof returns a string. TS typeof returns a type.

3 What does the satisfies operator do?

The satisfies operator validates that a value matches a type without widening it. Unlike a type annotation (: Type), which widens the inferred type, satisfies preserves the original literal types while still checking conformance.

This means you get both type safety (the value is checked against the broader type) and type precision (the inferred type keeps its specific literals). 

type Colors = "red" | "green" | "blue" | "black";

// With type annotation — type is widened to Record<string, Colors>
const palette1: Record<string, Colors> = {
  primary: "red",
  secondary: "blue",
};
palette1.primary; // type is Colors, NOT "red"

// With satisfies — preserves literal types!
const palette2 = {
  primary: "red",
  secondary: "blue",
} satisfies Record<string, Colors>;
palette2.primary; // type is "red" (literal preserved!)
💡 Memory Hook: satisfies = "check the type but keep the specifics". Annotation widens. Satisfies validates without losing detail.

4 What is the difference between an abstract class and an interface?

An abstract class provides both a contract and shared implementation (it produces runtime JavaScript code). An interface provides only a contract with no runtime code at all — it's completely erased during compilation.

Use interfaces for maximum flexibility (a class can implement many interfaces). Use abstract classes when you need shared default logic that subclasses can inherit. 

// Abstract class — has runtime code + contract
abstract class Animal {
  abstract name: string;
  move() { // shared implementation
    console.log(`${this.name} moves`);
  }
}

// Interface — contract only, no runtime code
interface Loggable {
  log(): void;
}

// A class can extend one abstract class
// but implement many interfaces
class Dog extends Animal implements Loggable {
  name = "Dog";
  log() { console.log(this.name); }
}
💡 Memory Hook: Abstract class = code + contract (runtime). Interface = contract only (compile-time only). Prefer interfaces for flexibility, abstract classes for shared logic.

5 What are decorators in TypeScript?

Decorators are functions that modify the behavior of classes, methods, properties, or parameters. They are an experimental feature in TypeScript — you must enable experimentalDecorators: true in your tsconfig.json.

Decorators are heavily used in frameworks like NestJS and Angular for dependency injection, route definitions, validation, and more. Think of them as "wrappers" that add metadata or behavior to a target. 

function Log(target: any, key: string, descriptor: PropertyDescriptor) {
  const original = descriptor.value;
  descriptor.value = function (...args: any[]) {
    console.log(`Calling ${key} with`, args);
    return original.call(this, ...args);
  };
}

class UserService {
  @Log
  getUser(id: number) {
    return { id, name: "Ali" };
  }
}
💡 Memory Hook: Decorators = "wrap this with extra behavior". They're functions that intercept and enhance classes, methods, and properties. Enable with experimentalDecorators.

6 What is function overloading in TypeScript?

Function overloading lets you declare multiple function signatures (overloads) with different parameter types, backed by a single implementation. The implementation signature must be compatible with all overloads.

This gives callers precise return types depending on what arguments they pass — much better than a single catch-all signature with any. 

// Overload signatures
function parse(input: string): number;
function parse(input: number): string;
// Implementation — must handle all overloads
function parse(input: string | number): string | number {
  if (typeof input === "string") return Number(input);
  return String(input);
}

const a = parse("42");  // type: number
const b = parse(42);    // type: string
💡 Memory Hook: Function overloading = different signatures, one implementation. The overloads describe what the caller sees. The implementation handles all cases internally.

✅ Correct! The implementation must accept all parameter combinations from the overload signatures.

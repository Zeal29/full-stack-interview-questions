Functions & Properties in TypeScript

TypeScript functions gain typed parameters, return types, and overloads — catching mismatched arguments at compile time. But functions aren't just about parameters and return values. TypeScript also provides powerful tools for defining and constraining object properties: making them optional, immutable, or derived from keys.

This lesson covers four essential concepts: optional and default parameters (flexible function signatures), readonly properties (immutability guarantees), keyof (dynamic key unions), and Record (typed dictionaries). Together, they let you write functions and objects that are both flexible and safe.

1 What are optional and default parameters?

TypeScript lets you mark function parameters as optional using ? — the caller can omit them, and their type implicitly includes undefined. You can also provide a default value using = value — if the caller passes undefined (or omits the argument), the default is used instead.

Rules: Optional parameters must come after required ones. Default parameters don't need to be last, but they become optional from the caller's perspective. 

// Optional parameter — can be undefined
function greet(name: string, title?: string): string {
  return title ? `${title} ${name}` : name;
}
greet("Ali");          // OK — title is undefined
greet("Ali", "Dr.");    // OK — "Dr. Ali"

// Default parameter — fallback value
function createUser(name: string, role: string = "viewer"): object {
  return { name, role };
}
createUser("Ali");              // { name: "Ali", role: "viewer" }
createUser("Ali", "admin");      // { name: "Ali", role: "admin" }

// ❌ Error: optional param before required
// function bad(title?: string, name: string) { }
💡 Memory Hook: ? = optional (might be undefined). = default = has fallback. Optional params must come after required ones.

2 What does readonly do?

The readonly modifier prevents reassignment of a property after initialization. Once set (in the constructor or object literal), the property cannot be changed. This is a compile-time check — JavaScript itself has no runtime enforcement.

Important caveat: readonly only prevents direct reassignment. If the property holds a nested object, that object's properties can still be mutated. For deep immutability, use Readonly<T> (shallow) or a library like immutable.js. 

interface Config {
  readonly apiKey: string;
  retries: number;
}

const config: Config = { apiKey: "abc123", retries: 3 };
config.retries = 5;       // OK
// config.apiKey = "xyz";  // ❌ Error: readonly

// ⚠️ Nested mutation still works!
interface State {
  readonly user: { name: string };
}
const s: State = { user: { name: "Ali" } };
s.user.name = "Sara";    // ⚠️ Allowed — nested mutation

// Use Readonly<T> for all properties at once
type FrozenConfig = Readonly<Config>;
💡 Memory Hook: readonly = write-once, like const but for properties. Only prevents direct reassignment — nested objects can still be mutated.

3 What is the keyof operator?

keyof creates a union type of an object type's keys. Given an interface or type alias, keyof T produces a string literal union of all its property names. It's essential for writing generic utility functions that need to reference property names in a type-safe way. 

interface User {
  name: string;
  age: number;
  email: string;
}

type UserKeys = keyof User;
// "name" | "age" | "email"

// Perfect for generic property access
function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user: User = { name: "Ali", age: 25, email: "ali@test.com" };
getProperty(user, "name");  // string ✅
// getProperty(user, "phone"); // ❌ Error — not a key of User
💡 Memory Hook: keyof = "give me all the keys as a union". Use it whenever you need to reference property names in a type-safe way.

4 What is Record<K, V>?

Record<K, V> is a built-in utility type that creates an object type with keys of type K and values of type V. It's TypeScript's way of defining a typed dictionary or map. Every key must be of type K, and every value must be of type V. 

// Any string key → number value
const scores: Record<string, number> = {
  alice: 95,
  bob: 87,
  charlie: 92,
};

// Combine with literal unions for strict keys
type Role = "admin" | "editor" | "viewer";
const permissions: Record<Role, string[]> = {
  admin: ["read", "write", "delete"],
  editor: ["read", "write"],
  viewer: ["read"],
};
// permissions.admin → string[] ✅
// permissions.superuser → ❌ Error — not a Role

// Equivalent to writing it out manually:
// { admin: string[]; editor: string[]; viewer: string[] }
💡 Memory Hook: Record = typed dictionary/map. Record<string, number> = any string key maps to a number. Combine with unions for strict keys.

🧪 Quick Check

1. What happens if an optional parameter is not provided?

✅ Correct! Optional parameters (marked with ?) become undefined when not provided by the caller.
❌ Optional parameters receive undefined when omitted. That's why their type implicitly includes undefined.

2. What does readonly prevent?

✅ Correct! readonly only prevents direct reassignment — nested objects can still be mutated.
❌ readonly only prevents direct reassignment of the property itself. Nested object properties can still be mutated.

3. What does keyof User produce if User has properties name, age, and email?

✅ Correct! keyof creates a string literal union of all property names — "name" | "age" | "email".
❌ keyof produces a union of the property names as string literals, not their values or an array.

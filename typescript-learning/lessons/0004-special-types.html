tsconfig.json — Your TypeScript Seatbelt

Every serious TypeScript project starts with a tsconfig.json file. It tells the compiler (tsc) how to behave — what JS version to target, which files to include, and how strict the type checking should be. The single most important option is strict: true, which enables all strict type-checking options at once.

Think of tsconfig.json as the project's TypeScript rulebook. Without it, you're running TypeScript in "loose mode" — and loose TypeScript misses the very bugs you adopted TypeScript to catch.

Special Types — void, never, unknown

TypeScript has three special types that serve distinct purposes. void marks functions that return nothing. never marks code paths that should never be reached. unknown is the type-safe alternative to any — you can assign anything to it, but you must narrow the type before using it. Understanding the difference between these three is a common interview topic and a practical daily skill.

Literal Types — Exactly This Value

Normally, a type describes a category of values — string means any string. Literal types narrow that down to one specific value. Combine them with unions to create precise, self-documenting constraints: type Direction = "left" | "right" | "up" | "down". The compiler will catch any typo or invalid assignment at build time.

1 What is tsconfig.json and what are its key options?

tsconfig.json is the configuration file for the TypeScript compiler. It tells tsc how to compile your project — which files to include, what JS version to output, and how strict the checking should be.

Key options:

  • strict: true — Enables all strict type-checking options (the most important one)
  • target — The JS version to compile to (e.g., "ES2020", "ESNext")
  • module — The module system (e.g., "commonjs", "ESNext")
  • outDir — Where to emit compiled JS files
  • include / exclude — Which files to compile or skip
{
  "compilerOptions": {
    "strict": true,
    "target": "ES2020",
    "module": "ESNext",
    "outDir": "./dist",
    "include": ["src/**/*"]
  }
}
💡 Memory Hook: Always enable strict: true — it's the seatbelt of TypeScript. Without it, you're driving without protection.

2 What is the void type?

void means a function returns nothing. It's the return type you assign when a function doesn't produce a value — it does work (like logging, saving, updating state) but doesn't hand anything back to the caller.

void is different from undefined. A function returning void can return undefined implicitly, but void communicates intent: "this function is not meant to return a value." 

function log(msg: string): void {
  console.log(msg);
}

function saveUser(user: User): void {
  db.save(user);
  // No return — just performs a side effect
}
💡 Memory Hook: void = "returns nothing". Think of it as a void — an empty space where a return value would be.

3 What is the never type?

never represents values that never occur. It's used in two scenarios: (1) functions that never return — they throw an error or loop forever, and (2) exhaustive type checking in switch statements, where the never branch should be unreachable.

If TypeScript infers never somewhere you didn't expect, it's usually telling you that your logic is contradictory — two types that can't overlap. 

// Function that never returns — always throws
function fail(msg: string): never {
  throw new Error(msg);
}

// Exhaustive switch checking
type Shape = "circle" | "square";

function area(shape: Shape): number {
  switch (shape) {
    case "circle": return Math.PI;
    case "square": return 1;
    default:
      // shape is `never` here — unreachable!
      const _exhaustive: never = shape;
      return _exhaustive;
  }
}
💡 Memory Hook: never = "impossible". If a value is never, something went wrong — or you're doing exhaustive checking.

4 What is the unknown type and how do you use it?

unknown is the type-safe counterpart of any. You can assign any value to an unknown variable, but you cannot use it without narrowing first. This forces you to prove what the type is before performing operations on it.

Narrowing techniques: typeof checks, instanceof, equality comparison, custom type guards, or type assertions. 

let value: unknown = fetchData();

// ❌ Can't use directly — TypeScript blocks this
value.toUpperCase(); // Error!

// ✅ Narrow with typeof
if (typeof value === "string") {
  console.log(value.toUpperCase()); // OK
}

// ✅ Narrow with instanceof
if (value instanceof Error) {
  console.log(value.message); // OK
}

// ✅ Custom type guard
function isString(v: unknown): v is string {
  return typeof v === "string";
}
💡 Memory Hook: unknown = "I don't know, prove it". Always prefer unknown over any — it gives you safety without sacrificing flexibility.

5 What are literal types?

Literal types restrict a variable to one exact value — a specific string, number, or boolean. You almost always combine them with union types to define a fixed set of allowed values.

This is far more precise than using string — the compiler will catch typos and invalid values at build time, not runtime. 

// String literal types
type Direction = "left" | "right" | "up" | "down";

function move(dir: Direction): void {
  // ...
}

move("left");   // ✅
move("diagonal"); // ❌ Error! Not in the union

// Number literal types
type DiceRoll = 1 | 2 | 3 | 4 | 5 | 6;

// Boolean literal types
type Truthy = true;
💡 Memory Hook: Literal types = "exactly this value". Not "a string" — but specifically "left" or "right".

Generics

Type variables, generic functions and classes, and type constraints

What You'll Learn

  • Write generic functions and classes with type parameters
  • Use generic constraints with extends

Understanding Generics

Generics let you write reusable, type-safe code that works with multiple types. Instead of writing separate functions for number, string, and boolean, you write one function with a type variable — a placeholder that the caller fills in.

The type variable <T> is like a function argument for types. You define it once, and every call site specifies the concrete type. TypeScript then checks that everything lines up — no runtime cost, full compile-time safety.

Basic Generic Function

The simplest generic — a function that returns whatever you pass in: 

function identity<T>(value: T): T {
  return value;
}

const a = identity<string>("hello");  // type: string
const b = identity(42);              // inferred as number

Generic Class

Classes can use generics too — the type variable is available throughout the entire class: 

class Box<T> {
  private contents: T;

  constructor(value: T) {
    this.contents = value;
  }

  getContents(): T {
    return this.contents;
  }
}

const stringBox = new Box("keys");     // Box<string>
const numberBox = new Box(99);        // Box<number>

Multiple Type Parameters

You're not limited to one type variable. Use as many as you need: 

function pair<A, B>(first: A, second: B): [A, B] {
  return [first, second];
}

const result = pair("age", 25);  // [string, number]

Generic Constraints with extends

Sometimes you need T to have certain properties. Use T extends SomeType to constrain what types are allowed: 

interface HasLength {
  length: number;
}

function logLength<T extends HasLength>(item: T): void {
  console.log(item.length);
}

logLength("hello");     // OK — string has .length
logLength([1, 2, 3]);     // OK — array has .length
logLength(123);           // Error — number has no .length

Default Type Parameters

You can provide a default type so callers don't have to specify it every time: 

interface PaginatedResponse<T = unknown> {
  data: T[];
  page: number;
  totalPages: number;
}

const fallback: PaginatedResponse = {  // T = unknown
  data: [],
  page: 1,
  totalPages: 0,
};

const users: PaginatedResponse<User> = { // T = User
  data: [{ name: "Ali" }],
  page: 1,
  totalPages: 5,
};

Common Built-in Generics

You've already been using generics without realizing it. TypeScript's built-in types are heavily generic: 

Array<T>           // T[] — an array of T
Promise<T>         // resolves to T
Record<K, V>       // object with keys K and values V
Map<K, V>          // key-value store
Partial<T>         // all properties of T optional
Omit<T, K>         // T without keys K

1 What are generics in TypeScript? (Q6)

Generics are type variables — placeholders for types that get filled in when you use the function, class, or interface. They let you write reusable, type-safe code that works with any type without sacrificing type information.

<T> is the most common type variable name (convention, not a keyword). You can name it anything — <TItem>, <TResult>, <K, V> for key-value pairs. 

// Without generics — loses type info
function identity(value: any): any {
  return value;
}

// With generics — preserves exact type
function identity<T>(value: T): T {
  return value;
}

const x = identity("hello");  // x: string (not any!)

Common generic patterns: Array<T>, Promise<T>, Record<K, V>, Map<K, V>, Partial<T>, Omit<T, K>

💡 Memory Hook: Generics = type variables. <T> is a placeholder for a type the user specifies when calling. Think of it like function arguments but for types — the caller decides the concrete type.

2 What are generic constraints and how do you use extends? (Q43)

Generic constraints limit what types a type variable can accept. T extends SomeType means "T must be at least this type" — it must have all the properties of SomeType.

The most powerful pattern is K extends keyof T — it constrains K to be a valid key of T. Combined with the return type T[K], you get fully type-safe property access: 

function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user = { name: "Ali", age: 25 };
const n = getProperty(user, "name");  // string
const a = getProperty(user, "age");   // number
getProperty(user, "email");          // Error! "email" not a key

Without the constraint, TypeScript couldn't guarantee that key actually exists on obj — you'd get runtime errors. With extends keyof T, invalid keys are caught at compile time.

💡 Memory Hook: extends in generics = "must be at least this type". K extends keyof T = "K must be a valid key of T". It's a compile-time guard that prevents invalid access.

🧪 Quick Check

1. What does <T> represent in a generic function?

✅ Correct! <T> is a type variable — like a function argument, but for types. The caller specifies the concrete type when using the function.
❌ <T> is a type variable (a placeholder). It's not a keyword, not a runtime value, and definitely not any — it preserves full type safety.

2. What does T extends HasLength enforce?

✅ Correct! T extends HasLength means T must have all properties of HasLength — in this case, a length: number property. It's structural, not inheritance-based.
❌ extends in generics is a constraint, not inheritance. It means "T must have at least these properties" — it's checked structurally at compile time.

3. What does K extends keyof T ensure?

✅ Correct! K extends keyof T constrains K to only valid keys of T. If T is {name: string, age: number}, K can only be "name" or "age".
❌ K extends keyof T ensures K is one of T's property keys. It's how TypeScript prevents typos and invalid property access at compile time.

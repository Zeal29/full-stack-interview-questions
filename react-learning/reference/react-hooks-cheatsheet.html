React Hooks Cheatsheet

Every React hook — signature, what it does, when to use it, and the common pitfall. Print-friendly.

Core: State & Effects

useState since 16.8

const [value, setValue] = useState(initialValue | () => initialValue)

Adds local state to a function component. Returns the current value and a setter. The initial value is used only on the first render; pass a function for lazy/expensive initialization.

When to use

For any local, component-owned data that should trigger a re-render when it changes.

Common pitfall

Stale updates — use the functional form setX(prev => ...) when the new value depends on the previous.

useEffect since 16.8

useEffect(() => { ... ; return cleanup }, [deps])

Runs side effects after render — data fetching, subscriptions, DOM manipulation, timers. The optional cleanup runs before the next effect and on unmount.

When to use

Syncing the component with an external system (network, DOM, subscription). Not for deriving data from props/state.

Common pitfall

Missing dependency → stale closure / infinite loop. Setting state without a dep array = infinite re-render.

useLayoutEffect since 16.8

useLayoutEffect(() => { ... ; return cleanup }, [deps])

Identical to useEffect but fires synchronously after DOM mutations, before the browser paints. Use to read layout and mutate state to avoid visible flicker.

When to use

Only when you must measure/mutate DOM before paint (tooltips, scroll positions, measurement-driven layout).

Common pitfall

Overuse — it blocks paint. Use useEffect by default; warns under SSR (use useIsomorphicLayoutEffect).

Refs & Identity

useRef since 16.8

const ref = useRef(initialValue)  // ref.current

Returns a mutable object whose .current persists across renders without triggering re-renders. Two uses: (1) access DOM nodes via the ref prop, (2) store any mutable value that shouldn't drive the UI.

When to use

DOM access (focus, scroll, measure), timers, "previous value" tracking, mutable flags.

Common pitfall

Treating it like state — mutating .current doesn't re-render. If the UI must update, use useState.

useImperativeHandle since 16.8

useImperativeHandle(ref, () => ({ method() {} }), [deps])

Customizes what a parent sees through a forwarded ref. Instead of exposing the raw DOM node, you expose only the methods you choose.

When to use

Reusable components that need a small imperative surface (focus, scroll, play). Pair with forwardRef.

Common pitfall

Exposing too much — it's an escape hatch from the declarative model. Expose the smallest API that works.

useId since 18

const id = useId()

Generates a unique, stable id that's consistent between server and client. Designed for accessibility attributes (label/input pairs).

When to use

Linking <label htmlFor> to <input id>; aria attributes that need a stable id across SSR.

Common pitfall

Using it for list keys or database ids — it's not for that. Keys come from your data; DB ids from the server.

Performance & Memoization

useMemo since 16.8

const memoized = useMemo(() => compute(a, b), [a, b])

Caches the result of a calculation, recomputing only when dependencies change. The value-level memoization hook.

When to use

Genuinely expensive computations, or when the memoized value is a dep of useEffect/memoized child.

Common pitfall

Over-memoizing trivial work — the memo overhead exceeds the saved computation. Measure first.

useCallback since 16.8

const memoizedFn = useCallback(() => { ... }, [deps])

Caches a function reference (the function counterpart to useMemo) so it keeps identity across renders unless deps change.

When to use

Only when the function's identity is consumed — as a useEffect dep or a prop to a React.memo child.

Common pitfall

Wrapping every function "for perf" — it adds overhead with no benefit unless something compares the reference.

useDebugValue since 16.8

useDebugValue(value, formatFn?)

Displays a label for a custom hook in React DevTools. Optional formatter avoids running expensive formatting unless DevTools is open.

When to use

Inside custom hooks to make their internal state readable in DevTools.

Common pitfall

Calling it in components (not hooks) — it's a hook, follows the rules of hooks.

State Management & Context

useReducer since 16.8

const [state, dispatch] = useReducer(reducer, initialArg, init?)

Alternative to useState for complex state: you dispatch actions, a pure reducer computes the next state. "Redux-lite."

When to use

Multiple interdependent fields, complex transitions, state machines, or testable state logic.

Common pitfall

Reaching for it for single-value state — that's useState territory. Keep reducers pure.

useContext since 16.8

const value = useContext(SomeContext)

Reads the nearest Provider's value for a context. The component re-renders whenever that value changes (by reference).

When to use

App-wide, low-frequency data (theme, auth, locale) to avoid prop drilling.

Common pitfall

High-frequency updates cause all consumers to re-render. Split contexts or use an external store.

Concurrent React

useTransition since 18

const [isPending, startTransition] = useTransition()

Marks a state update as a non-urgent transition. Urgent updates (typing) stay responsive; the transition can be interrupted. isPending flags in-progress work.

When to use

Expensive updates triggered by user input (filtering large lists, tab switches) where you want a loading flag.

Common pitfall

Wrapping urgent updates (the input value itself) — those must stay urgent or typing feels laggy.

useDeferredValue since 18

const deferred = useDeferredValue(value)

Returns a deferred copy of a value that lags behind urgent updates, refreshing when the browser is idle. The value-level counterpart to useTransition.

When to use

When the value comes from props/a library and you can't wrap the setter — feed the deferred value to expensive work.

Common pitfall

Using it everywhere — it only helps when downstream work is genuinely expensive and the value changes often.

useSyncExternalStore since 18

const snapshot = useSyncExternalStore(subscribe, getSnapshot, getServerSnapshot?)

The concurrent-safe way to subscribe to a non-React store (Redux, Zustand, browser APIs). Prevents "tearing" — inconsistent snapshots across a render.

When to use

Authoring a store adapter or bridging browser APIs (online status, media queries) to React under concurrent rendering.

Common pitfall

Returning a new object from getSnapshot each call — React compares by reference and will loop. Return a cached snapshot.

React 19

use since 19

const value = use(promise | context)

Reads a Promise or Context during render. Suspends the component (shows nearest Suspense fallback) until a promise resolves. The only hook you can call conditionally or in a loop.

When to use

Reading a promise passed from a Server Component or cache; reading context conditionally. The React 19 data model.

Common pitfall

Creating a new promise each render — pass a stable promise (from props/memo) or you'll suspend forever.

useFormStatus since 19

const { pending, data, method, action } = useFormStatus()

Gives a component the pending state of its parent <form>'s action — without prop-drilling a loading flag. Must be a descendant of a form.

When to use

Building submit buttons that show a pending state for Server Actions / form actions.

Common pitfall

Calling it in the form itself — it reads the parent form, so it must be a child component.

useOptimistic since 19

const [optimisticState, addOptimistic] = useOptimistic(state, reducer?)

Shows an optimistic update instantly during an async operation and auto-reverts on failure or reconciles to the real value on success.

When to use

Actions with perceived latency — likes, comments, toggles — where instant feedback improves UX.

Common pitfall

Forgetting it reverts on error — design your UI to handle the revert (don't assume success).

useActionState since 19

const [state, formAction, isPending] = useActionState(actionFn, initialState)

Manages state for a form action: holds the result of the action, exposes a pending flag, and returns a formAction you wire to <form action>.

When to use

Forms with Server Actions that return a result (validation errors, success message) you want to display.

Common pitfall

Mixing it up with useFormStatus — this manages the action's result; that manages the form's pending state.

React Component Patterns

When to reach for each pattern, a code skeleton, and — just as important — when not to.

1. Higher-Order Component (HOC) cross-cutting

A function that takes a component and returns an enhanced component. The classic pre-hooks way to share cross-cutting logic.

const withAuth = (Component) => {
  return function Protected(props) {
    const user = useAuth();
    if (!user) return <Login />;
    return <Component {...props} user={user} />;
  };
};

// Usage
const Dashboard = withAuth(RawDashboard);

✓ When to use

  • Cross-cutting concerns that wrap the whole component (auth gating, telemetry injection).
  • Interoperating with libraries built on HOCs (older connect() redux).

✗ When NOT to use

  • For simple logic sharing — a custom hook is almost always simpler and avoids "wrapper hell."
  • When you need fine-grained prop control — render props or hooks are clearer.

2. Render Props data + render separation

A prop whose value is a function returning JSX. The parent owns the data; the consumer decides how to render it.

function MouseTracker({ children }) {
  const [pos, setPos] = useState({ x: 0, y: 0 });
  // ...track mouse, update pos
  return children(pos);
}

// Consumer decides rendering:
<MouseTracker>
  {({ x, y }) => <div>Cursor at {x}, {y}</div>}
</MouseTracker>

✓ When to use

  • When the consumer must fully control the rendered output of shared logic.
  • Working with older libraries (react-router v5, react-motion).

✗ When NOT to use

  • New code — a custom hook (useMousePosition()) is simpler and avoids the extra nesting.
  • Pure rendering logic — just compose components.

3. Custom Hooks logic reuse (modern default)

A function starting with use that extracts reusable stateful logic. The modern replacement for HOCs and render props.

function useFetch(url) {
  const [state, setState] = useState({ data: null, error: null, loading: true });

  useEffect(() => {
    let ignore = false;
    setState(s => ({ ...s, loading: true }));
    fetch(url)
      .then(r => r.json())
      .then(data => { if (!ignore) setState({ data, error: null, loading: false }); })
      .catch(error => { if (!ignore) setState(s => ({ ...s, error, loading: false })); });
    return () => { ignore = true; };
  }, [url]);

  return state;
}

✓ When to use

  • Sharing stateful logic between components — the default choice in modern React.
  • Extracting a non-trivial effect/state sequence out of a component for clarity.

✗ When NOT to use

  • For pure, stateless helpers — use a plain function (no use prefix).
  • Returning JSX — that's a component, not a hook.

4. Compound Components composable API

A set of components sharing state implicitly through context, exposing an API that reads like native HTML. Powers design-system primitives.

const TabsContext = createContext();

function Tabs({ children, defaultIndex }) {
  const [active, setActive] = useState(defaultIndex ?? 0);
  return (
    <TabsContext.Provider value={{ active, setActive }}>
      {children}
    </TabsContext.Provider>
  );
}

Tabs.Tab = function Tab({ index, children }) {
  const { active, setActive } = useContext(TabsContext);
  return <button onClick={() => setActive(index)} aria-selected={active === index}>
    {children}
  </button>;
};

✓ When to use

  • Building design-system primitives (Tabs, Accordion, Menu, Dialog) where consumers need flexible composition.
  • When sub-components must share implicit state without prop drilling.

✗ When NOT to use

  • Simple, fixed-layout components — a single prop-driven component is clearer.
  • When the API surface is tiny — the context boilerplate isn't worth it.

5. Context + Provider Pattern dependency broadcast

Broadcast a value to a subtree without prop drilling. Pair with a custom hook consumer for a clean API.

// 1. Create + Provider wrapper
const ThemeContext = createContext(null);

export function ThemeProvider({ children }) {
  const [theme, setTheme] = useState("light");
  const value = useMemo(() => ({ theme, setTheme }), [theme]);
  return <ThemeContext.Provider value={value}>{children}</ThemeContext.Provider>;
}

// 2. Consumer hook (throws if used outside provider)
export function useTheme() {
  const ctx = useContext(ThemeContext);
  if (!ctx) throw new Error("useTheme must be inside ThemeProvider");
  return ctx;
}

✓ When to use

  • App-wide, low-frequency dependencies (theme, auth, locale, feature flags).
  • Providing a service implementation (so tests can swap it).

✗ When NOT to use

  • High-frequency updates — every consumer re-renders on value change.
  • As a Redux replacement for large app state — use an external store with slice subscriptions.

6. forwardRef + useImperativeHandle imperative escape hatch

Expose a controlled imperative API (focus, scroll, play) from a child to its parent, while hiding the raw DOM node.

const Video = forwardRef((function Video(props, ref) {
  const videoRef = useRef();

  useImperativeHandle(ref, () => ({
    play:  () => videoRef.current.play(),
    pause: () => videoRef.current.pause(),
    seek:  (t) => { videoRef.current.currentTime = t; },
  }), []);

  return <video ref={videoRef} {...props} />;
}));

// Parent gets a tiny imperative API, not the whole <video> node
<Video ref={playerRef} />;
playerRef.current.seek(30);

✓ When to use

  • Reusable components that need a small imperative surface (focus, scroll, media control).
  • Bridging React with imperative DOM APIs (canvas, video, third-party widgets).

✗ When NOT to use

  • If the parent can express the same thing declaratively via props — prefer that.
  • Exposing the whole DOM node — encapsulate; expose only what's needed.

7. Error Boundary resilience

Catch render-phase errors in a subtree and show fallback UI. Must be a class component. Wrap independent sections so a single crash doesn't blank the page.

class ErrorBoundary extends React.Component {
  state = { error: null };

  static getDerivedStateFromError(error) {
    return { error };
  }

  componentDidCatch(error, info) {
    logToTelemetry(error, info);
  }

  render() {
    if (this.state.error) return this.props.fallback(this.state.error);
    return this.props.children;
  }
}

// Usage — wrap risky subtrees
<ErrorBoundary fallback={() => <Oops />}>
  <RiskyWidget />
</ErrorBoundary>

✓ When to use

  • Anywhere a child crash shouldn't take down the whole app (panels, widgets, routes).
  • Pairing with Suspense — boundary inside Suspense catches errors + suspends.

✗ When NOT to use

  • For event-handler or async errors — those need try/catch / promise catches.
  • Don't wrap the entire app in one boundary — isolate failures.

8. Suspense Boundary Structure loading orchestration

Wrap async units (lazy components, use(promise)) in their own Suspense + Error Boundary so they load and fail independently.

// One boundary per independent async unit
<Layout>
  <Header /> // instant — outside any Suspense

  <ErrorBoundary fallback={<Error />}>
    <Suspense fallback={<Spinner />}>
      <Panel dataPromise={p1} />
    </Suspense>
  </ErrorBoundary>

  <ErrorBoundary fallback={<Error />}>
    <Suspense fallback={<Spinner />}>
      <Chart dataPromise={p2} />
    </Suspense>
  </ErrorBoundary>
</Layout>

✓ When to use

  • Code-splitting large routes/components (React.lazy).
  • The React 19 data model — components reading promises with use.
  • SSR streaming — flush the shell, stream sections as data resolves.

✗ When NOT to use

  • One giant Suspense around the whole page — it blocks everything on the slowest child.
  • Without an Error Boundary nearby — a rejected promise will crash the tree.

9. useReducer State Machine complex transitions

Model a component's state as explicit states + transitions with a typed action union. Makes complex logic testable and transition-safe.

type State =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T }
  | { status: "error"; error: Error };

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case "FETCH":  return { status: "loading" };
    case "SUCCESS": return { status: "success", data: action.data };
    case "ERROR":  return { status: "error", error: action.error };
    case "RESET":  return { status: "idle" };
  }
}

const [state, dispatch] = useReducer(reducer, { status: "idle" });

✓ When to use

  • Multi-step flows (fetch → loading/success/error), forms with validation rules, wizards.
  • When you want transition rules enforced ("can't go from error to success directly").
  • When the logic needs unit tests — reducers are pure.

✗ When NOT to use

  • Single independent value — useState is simpler.
  • App-wide state with many consumers — an external store (Zustand/Redux) scales better.

The Concurrent Renderer

This is where interviews separate juniors from seniors. React 16 rewrote the engine as Fiber — a unit-of-work architecture that can pause, resume, and prioritize rendering. React 18 built on Fiber to deliver concurrent rendering: React can interrupt a slow render to handle an urgent update (a keystroke), then resume the slow work later. The user-facing APIs for this are startTransition, useTransition, and useDeferredValue.

Two performance primitives underpin everything else: reconciliation (the diffing algorithm — "what changed?") and batching (collapsing multiple state updates into one re-render). Get these mental models right and you can reason about any performance question from first principles.

1What is the difference between useEffect and useLayoutEffect?

Timing. useEffect runs asynchronously, after the browser paints — non-blocking, so it's the right default for almost everything (fetches, subscriptions, analytics). useLayoutEffect runs synchronously, before the browser paints — it blocks the paint, so the user never sees the unmodified layout. 

// useLayoutEffect: measure then fix layout before paint (no flicker)
useLayoutEffect(() => {
  const h = ref.current.getBoundingClientRect().height;
  setTooltipPos(computePos(h));
}, [dep]);

Use useLayoutEffect only when you must read DOM layout and synchronously mutate state to avoid a visible flicker (tooltips, measuring before animating). Reach for it after useEffect proves insufficient. Server-side, neither runs — but useLayoutEffect warns; use useIsomorphicLayoutEffect if you SSR.

💡 Memory Hook: Use useEffect 99% of the time. Use useLayoutEffect only to modify DOM before the user sees it (prevent flicker on measurement-driven layout).

2What is reconciliation in React?

Reconciliation is React's algorithm for diffing the old and new virtual DOM trees to decide the minimum set of real-DOM mutations. It runs three rules:

  1. Different element types → tear down the old subtree and build a new one (e.g. <a> → <div> destroys everything).
  2. Same element type → keep the DOM node, update only changed props/attributes recursively.
  3. Children → matched by key across renders (this is why stable keys matter — Lesson 3).
// Type change → full rebuild
{isLink ? <a /> : <div />}

// Same type → update props in place
<Profile user={u1} />
<Profile user={u2} />

The heuristic is O(n) rather than the theoretical O(n³) of a full tree diff, which is what makes it fast enough. This is why you should not swap component types to "reset state" casually — it forces a full subtree rebuild.

💡 Memory Hook: Reconciliation = "what changed?" Different type = nuke and rebuild. Same type = update in place. Keys = identity for list children.

3What are Suspense and lazy loading?

React.lazy + Suspense enable code splitting: load a component's code only when it's first rendered, shrinking the initial JS bundle. Suspense shows a fallback (spinner, skeleton) while the lazy component's chunk loads. 

const HeavyChart = React.lazy(() => import("./HeavyChart"));

<Suspense fallback={<Spinner />}>
  <HeavyChart />
</Suspense>

Place Suspense boundaries thoughtfully: too high and one slow chunk blocks a whole page; too low and you get janky partial loads. Suspense also integrates with data fetching (via frameworks like Next.js or libraries like Relay) — a component that suspends triggers the nearest Suspense fallback.

💡 Memory Hook: lazy + Suspense = load on demand. Split large routes/components. Always provide a fallback — without it, React throws if the chunk isn't ready.

4What is the React Fiber architecture?

Fiber is React's reconciliation engine, rewritten in React 16. Before Fiber, React rendered synchronously and couldn't be interrupted — a huge update would block the main thread. Fiber breaks rendering work into units called "fibers", each linked into a tree, that can be paused, resumed, and prioritized.

  • Incremental rendering — work splits into chunks; React can yield to the browser between them.
  • Priority-based updates — user input (urgent) preempts data fetching (non-urgent).
  • Foundation for concurrent features — startTransition, useDeferredValue, server components.

You won't write Fiber code directly, but knowing it exists explains why transitions and deferred values work: they're priorities handed to the Fiber scheduler, not debounce tricks.

💡 Memory Hook: Fiber = React's new engine (since v16). Breaks work into interruptible units, supports priorities. The foundation under all concurrent features.

5How does React's batching work?

Batching collapses multiple state updates into a single re-render. Three setState calls in one handler produce one render, not three. This avoids wasteful intermediate paints and is the reason the functional-updater pattern matters (Lesson 2): within a batched update, each call sees the same starting state. 

// React 18+ — ALL of these batch into ONE re-render
function handleClick() {
  setCount(1);
  setName("Ali");
  setActive(true);
  // → single render here
}

// Batches everywhere: promises, timeouts, native events too
setTimeout(() => { setA(1); setB(2); }, 100); // one render

Before React 18, only updates inside React event handlers were batched — updates in setTimeout, promises, or native listeners each caused a separate render (a common perf footgun). React 18's automatic batching batches everywhere. The escape hatch is flushSync (later card), which forces an immediate, synchronous update.

💡 Memory Hook: React 18 = automatic batching everywhere. Multiple setState calls = one render. Use flushSync to opt out only when you genuinely need synchronous DOM update.

6What is startTransition?

startTransition marks a state update as low-priority (non-urgent). Urgent updates (typing in an input, clicks) keep the UI responsive; the transition update is deferred and can be interrupted by a newer urgent update. 

const [search, setSearch] = useState("");
const [filter, setFilter] = useState("");

function handleChange(e) {
  setSearch(e.target.value);        // urgent — input stays snappy
  startTransition(() => {
    setFilter(e.target.value);      // non-urgent — expensive list filter
  });
}

Classic use case: a search box filtering thousands of rows. Without a transition, every keystroke blocks on the expensive filter; with it, typing stays instant and the list catches up. Note: startTransition doesn't give you a pending flag — for that, use useTransition (next card).

💡 Memory Hook: startTransition = "this update can wait." Split urgent (input) from non-urgent (filtering) updates to keep the UI responsive.

7What is useDeferredValue?

useDeferredValue returns a deferred copy of a value that lags behind the urgent one. The deferred value updates when the browser is idle, so expensive work that depends on it doesn't block urgent updates. 

const [search, setSearch] = useState("");
const deferredSearch = useDeferredValue(search);

// Heavy list uses the deferred value:
const results = useMemo(
  () => heavyFilter(allItems, deferredSearch),
  [allItems, deferredSearch]
);

It's the value-level counterpart to startTransition (which is update-level). Reach for useDeferredValue when you don't control where the value is set (it comes from props or a library) — startTransition when you do.

💡 Memory Hook: useDeferredValue = "use the old value until you're free." Like debouncing, but driven by browser availability rather than a fixed time delay.

8What is the useTransition hook?

useTransition is the hook version of startTransition. It returns a tuple [isPending, startTransition]: wrap your expensive update in startTransition, and use isPending to show a loading indicator while the transition is processing. 

const [isPending, startTransition] = useTransition();

function onTabChange(id) {
  startTransition(() => {
    setActiveTab(id); // expensive re-render of tab content
  });
}

return (
  <button onClick={onTabChange}>
    {isPending ? "Loading…" : "Switch tab"}
  </button>
);

Prefer useTransition over bare startTransition whenever you want a pending flag. The pending state lets you give the user feedback (dimming, spinner) without blocking their input.

💡 Memory Hook: useTransition = non-urgent update + a loading flag. isPending tells you if the transition is still processing — use it for inline spinners.

9What is the useSyncExternalStore hook?

useSyncExternalStore is the safe way to subscribe a component to a non-React store (Redux, Zustand, browser APIs like matchMedia, online status) under concurrent rendering. It takes a subscribe function, a snapshot function, and an optional server snapshot. 

const isOnline = useSyncExternalStore(
  (cb) => {            // subscribe
    window.addEventListener("online", cb);
    return () => window.removeEventListener("online", cb);
  },
  () => navigator.onLine, // getSnapshot (client)
  () => true               // getServerSnapshot
);

It exists to prevent "tearing" — where concurrent rendering would let different parts of the UI read different snapshots of an external store within a single render pass. Before React 18, libraries hacked this with useMutableSource; now there's a single blessed API. As an app developer you rarely write it yourself — your state library does.

💡 Memory Hook: useSyncExternalStore = safe external-data bridge. Required when connecting non-React state to React in concurrent mode — prevents tearing.

10What is flushSync?

flushSync forces React to flush a state update to the DOM synchronously, overriding automatic batching. After flushSync(fn) returns, the DOM is guaranteed to reflect the updates inside fn. 

import { flushSync } from "react-dom";

flushSync(() => {
  setScrollTarget(id);
});
// DOM is updated here — safe to measure / scroll
element.scrollIntoView();

Use it sparingly — it defeats batching, which is usually a perf loss. Legit cases: you must measure or scroll the DOM immediately after an update; integrating with non-React code that needs the DOM in a known state; flushing before showing a "beforeunload" prompt. If you find yourself reaching for it often, you've usually mis-modelled something.

💡 Memory Hook: flushSync = "update NOW, no batching." Escape hatch. Use only when you absolutely need synchronous DOM updates (measure/scroll/imperative integration).

🧪 Quick Check

1. When does useLayoutEffect fire relative to useEffect?

✅ Correct! useLayoutEffect runs synchronously before paint (use it to avoid flicker); useEffect runs asynchronously after paint.
❌ useLayoutEffect runs synchronously before the browser paints; useEffect runs after. Use the layout version only to prevent visible flicker.

2. In React 18, what happens to three setState calls inside a setTimeout?

✅ Correct! React 18's automatic batching applies everywhere — timeouts, promises, native handlers — not just React events.
❌ React 18 batches updates everywhere (timeouts, promises, native events), collapsing them into one render. Pre-18, these would have caused three renders.

3. What problem does useSyncExternalStore solve?

✅ Correct! It's the concurrent-safe bridge to non-React stores — guarantees a consistent snapshot across the render, preventing tearing.
❌ Its job is to prevent tearing: under concurrent rendering, different subtrees could otherwise read different snapshots of an external store. useSyncExternalStore makes that impossible.

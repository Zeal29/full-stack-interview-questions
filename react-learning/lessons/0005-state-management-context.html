State Management & Context

Prop drilling, the Context API, and useReducer — choosing where state lives and how it flows

What You'll Learn

  • Identify prop drilling and know when to refactor away from it
  • Use the Context API correctly, including its re-render caveat
  • Choose useReducer over useState for complex state
  • Reason about state colocation and the right place to hold data

Where Should State Live?

The single most important state-management principle is colocation: keep state as close to where it's used as possible. Lift it up only when siblings need to share it; broadcast it globally only when it's truly app-wide (theme, auth, locale). Most state belongs in a single component or a small subtree — not in a global store.

When you do need to share data across a deep tree, you have three tools in increasing order of power: prop drilling (fine for 1-2 levels), Context (broadcast to a subtree without re-passing props), and useReducer (manage complex state transitions). An external library (Zustand, Redux, Jotai) is the escape hatch when Context's re-render behaviour becomes a bottleneck.

1What is prop drilling?

Prop drilling is passing props through intermediate components that don't use them, just to reach a deeply nested child. The middle components become pure forwarders — they know about data they don't need, which couples them to the deep child's requirements. 

// theme threaded through Layout and Sidebar just to reach Button
<App theme="dark">
  <Layout theme={theme}>
    <Sidebar theme={theme}>
      <Button theme={theme} />

Solutions: Context API (next card), a state library (Zustand/Redux/Jotai), or component composition — passing children as JSX so intermediate components don't need to know about the data. A little drilling (1-2 levels) is fine and often clearer than a context; reach for the bigger tools when it's 3+ levels or the middle components have no business knowing the data.

💡 Memory Hook: Prop drilling = passing the baton through people who don't need it. If 3+ intermediates just forward a prop, use Context, composition, or a state library.

2What is the Context API?

Context provides a way to share values across a subtree without prop drilling. Create a context, wrap consumers in a Provider with a value, and any descendant can read it via useContext — no matter how deep. 

const ThemeContext = createContext("light");

<ThemeContext.Provider value="dark">
  <App />
</ThemeContext.Provider>

// Any descendant:
const theme = useContext(ThemeContext);

Best for: low-frequency, app-wide data — theme, auth user, locale, feature flags. The re-render caveat: every consumer re-renders whenever the Provider's value changes (by reference). That makes Context a poor fit for high-frequency updates (e.g. a fast-changing slider value consumed widely) — split contexts, memoize the value object, or reach for an external store that uses useSyncExternalStore.

💡 Memory Hook: Context = broadcasting — one provider, many consumers anywhere below. Use for truly global data (theme, auth). Not for high-frequency updates — every consumer re-renders on value change.

3What is the useReducer hook?

useReducer is an alternative to useState for complex state logic. Instead of "set this value", you dispatch an action describing what happened, and a pure reducer function computes the next state. It shines when state involves multiple interdependent fields or when the next state depends on the previous in non-trivial ways. 

const [state, dispatch] = useReducer(reducer, initialState);

function reducer(state, action) {
  switch (action.type) {
    case "increment": return { ...state, count: state.count + 1 };
    case "reset":     return initialState;
    default:           return state;
  }
}
// dispatch({ type: "increment" })

Reach for it when: you have many useStates that change together (a multi-field form, a stepper, a fetch state machine), when transitions have rules (valid/invalid moves), or when you want testable state logic (reducers are pure functions). It's "Redux-lite" without the library.

💡 Memory Hook: useReducer = Redux-lite. Use when useState gets messy (multiple interdependent values, complex transitions). Flow: dispatch → reducer → new state.

🧪 Quick Check

1. When a Context Provider's value prop changes, what happens to consumers?

✅ Correct! A value change (by reference) re-renders all consumers. That's why Context fits low-frequency data, not fast-changing values.
❌ A new value reference re-renders every consumer of that context. High-frequency updates in context cause widespread re-renders.

2. Which is the BEST fit for useReducer over useState?

✅ Correct! Multiple interdependent fields with transition rules are exactly what reducers model cleanly and testably.
❌ useReducer earns its keep with many interdependent values and complex transitions. Single-value state is useState territory.

3. What's the FIRST fix to try for prop drilling before reaching for Context?

✅ Correct! Composition (children as JSX) often removes drilling entirely — the parent owns the data and passes the ready-made subtree, so intermediates stay decoupled.
❌ Composition is the cheapest fix: the data owner passes children as JSX, so intermediates never see the data. Context/store come next when that's not enough.

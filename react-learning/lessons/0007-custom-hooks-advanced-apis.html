Logic Reuse and the Imperative Bridge

Custom hooks are the modern answer to "how do I share stateful logic between components?" — a function starting with use that can call other hooks. They replaced HOCs and render props for logic reuse. A few rules: the name must start with use (so the linter's rules-of-hooks apply), and they share behaviour, not state — each call site gets its own state.

The other APIs here bridge React's declarative world and the imperative one: useId generates stable ids for label/input pairs across server and client; forwardRef lets a parent pass a ref through to a child's DOM node; useImperativeHandle customises what that ref exposes. And StrictMode is the dev-only safety net that double-invokes renders and effects to surface bugs.

1What is the useId hook?

useId generates a unique, stable id that's consistent between server and client renders. It's meant for accessibility attributes that link elements — most commonly <label htmlFor> paired with <input id>. 

function Field() {
  const id = useId();
  return (
    <>
      <label htmlFor={id}>Name</label>
      <input id={id} />
    </>
  );
}

Don't use it for: list keys (use stable ids from your data), database keys, or anything that needs to be globally sequential. The ids it produces are stable but not sequential — they look like :r0:, :r1:.

💡 Memory Hook: useId = SSR-safe unique ids for accessibility (label/input pairs). Not for list keys or DB ids.

2What are custom hooks?

Custom hooks are functions that extract reusable, stateful logic out of components. By convention the name must start with use — that's how the linter applies the rules of hooks. They can call other hooks, return values (state, functions, arrays — anything), and each call site maintains its own independent state. 

function useLocalStorage(key, initialValue) {
  const [value, setValue] = useState(() => {
    const stored = localStorage.getItem(key);
    return stored ? JSON.parse(stored) : initialValue;
  });

  useEffect(() => {
    localStorage.setItem(key, JSON.stringify(value));
  }, [key, value]);

  return [value, setValue];
}

They replaced HOCs and render props for logic reuse: no wrapper components, no nesting, just a function. The golden rule: share behaviour, not state — two components calling useLocalStorage("theme") each get their own copy.

💡 Memory Hook: Custom hooks = reusable stateful logic. Must start with use. Share behaviour, not state — each call site is independent. They replaced HOCs/render props.

3What is StrictMode in React?

<React.StrictMode> is a development-only wrapper that surfaces potential problems. It detects unsafe legacy lifecycle methods, warns about deprecated APIs, and — importantly — double-invokes render and effects in dev to flush out bugs caused by assuming a function runs once. 

<React.StrictMode>
  <App />
</React.StrictMode>

The double-invocation is the part that surprises people: in dev, your component body, useState initializers, and effects each run twice. This is not a bug — it deliberately exposes code that relies on side effects happening exactly once (which breaks under React's concurrent renderer, where work can be discarded and re-run). It has zero production effect.

💡 Memory Hook: StrictMode = dev safety net. Double-invokes render/effects in dev to catch impure code. No impact on production. If something runs twice and breaks, your code wasn't pure.

4What is forwardRef?

forwardRef lets a parent pass a ref through to a child component's DOM node. By default, ref is a special prop that React handles — it does not show up in the child's props. forwardRef opts in: it gives the child a second argument ref that you can attach to a DOM element. 

const Input = React.forwardRef((props, ref) => (
  <input ref={ref} {...props} />
));

// Parent can now grab the input's DOM node:
<Input ref={inputRef} />
inputRef.current.focus();

(In React 19, ref is a regular prop for function components and forwardRef is being soft-deprecated — but you'll see it everywhere in existing code and libraries.) Use it for inputs you want to focus, scroll containers you want to measure, and when building design-system primitives.

💡 Memory Hook: forwardRef = "pass the ref through." Without it, ref is consumed by React and never reaches the child. Used for input focus, scroll, and imperative handles.

5What is useImperativeHandle?

useImperativeHandle customizes the instance value exposed to the parent via ref. By default, forwarding a ref gives the parent the raw DOM node. With useImperativeHandle, you expose only specific methods — a controlled, encapsulated imperative API. 

const Input = React.forwardRef((props, ref) => {
  const inputRef = useRef();

  useImperativeHandle(ref, () => ({
    focus: () => inputRef.current.focus(),
    clear: () => { inputRef.current.value = ""; },
  }));

  return <input ref={inputRef} />;
});

// Parent calls ref.current.focus() — only what you exposed, not the DOM node

Use it sparingly — it's an escape hatch from React's declarative model. The right use case is reusable components that need a small, well-defined imperative surface (focus, scroll, play/pause a video). Avoid exposing the entire DOM node; expose the smallest API that does the job.

💡 Memory Hook: useImperativeHandle = "here's what I'll let the parent do." Encapsulates the ref — expose only specific methods, not the whole DOM node.

🧪 Quick Check

1. Why must a custom hook's name start with use?

✅ Correct! The use prefix is how the eslint-plugin-react-hooks linter knows to apply the rules of hooks. Skip it and violations go uncaught.
❌ The use prefix tells the rules-of-hooks linter to enforce correct usage inside the function. It's a real tooling contract, not a style choice.

2. Two components both call useLocalStorage("theme", "light"). How many separate states exist?

✅ Correct! Custom hooks share behaviour, not state. Each call site is a fresh useState — independent React state, even if they read/write the same localStorage key.
❌ Custom hooks share behaviour, not state. Each call site runs its own useState — two independent states (which happen to both touch localStorage).

3. In dev with StrictMode, your effect's cleanup seems to run twice. This is…

✅ Correct! StrictMode intentionally mounts, unmounts, and remounts in dev to catch missing cleanups. It never ships to production.
❌ StrictMode intentionally double-invokes effects in dev to surface bugs (missing cleanups, side effects that assume one run). It has no production effect.

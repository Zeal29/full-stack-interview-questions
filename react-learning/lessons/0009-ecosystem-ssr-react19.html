React on the Server and Beyond

The biggest shift since hooks is the move back to the server. React Server Components (RSC) run on the server, can be async, and ship zero JavaScript to the client. They pair with Server Actions (form handlers that run on the server, no API routes needed) and a refreshed data model. Next.js's App Router is the canonical implementation, but RSC is a React feature, not a Next feature.

React 19 also added ergonomics hooks that were previously painful: useFormStatus (form pending state without prop drilling), useOptimistic (instant UI updates that auto-revert on failure), useActionState (form action state machine), and the use hook (read a promise or context during render — the only hook you can call conditionally). And the React Compiler aims to make manual useMemo/useCallback a thing of the past.

1What are Server Components (RSC)?

React Server Components run on the server and send rendered output (not their source) to the client. They can be async and fetch data or touch the filesystem directly — and they add zero JavaScript to the bundle, because their code never reaches the browser. Client Components opt in with the "use client" directive. 

// Server Component (default in the App Router)
async function ArticleList() {
  const articles = await db.query("SELECT * FROM articles");
  return articles.map(a => <Article key={a.id} {...a} />);
}

The mental model: Server Components fetch and render static or data-driven UI; Client Components handle interactivity (state, effects, event handlers). You compose them — a Server Component can render a Client Component, but a Client Component can't directly render a Server Component (it can via children passed down).

💡 Memory Hook: Server Components = zero-JS backend rendering. async component = fetch data directly. "use client" = opt into client-side rendering (state/effects).

2What is the difference between React and ReactDOM?

react is the core, platform-agnostic library — it defines components, hooks, elements, and the reconciliation algorithm. It knows nothing about the DOM. react-dom is the browser/DOM adapter — it provides createRoot, hydrateRoot, event handling, and turns React elements into real DOM nodes. 

// react: define UI
import { useState } from "react";

// react-dom: render to the browser
import { createRoot } from "react-dom/client";
createRoot(document.getElementById("root")).render(<App />);

This split lets the same React core target other platforms: react-native (mobile), react-three-fiber (WebGL), react-terminal (CLI), ink (terminal). Same components and hooks — different "renderer".

💡 Memory Hook: react = brain (logic, hooks, reconciliation). react-dom = hands (DOM manipulation). Same brain, different hands for web vs native.

3What is hydration?

Hydration is the process where React "adopts" server-rendered HTML on the client — attaching event listeners and reconciling the existing DOM instead of throwing it away and rebuilding. The server sends fully-rendered HTML (fast first paint, SEO-friendly); the client then makes it interactive. 

// Client entry for an SSR app
import { hydrateRoot } from "react-dom/client";
hydrateRoot(document.getElementById("root"), <App />);

If the server HTML and the client's first render disagree (a hydration mismatch), React warns in dev and re-renders that subtree from scratch — a perf hit. Common causes: using Date.now() or Math.random() in render, or reading browser-only APIs during SSR. React 18+ supports selective hydration — it can hydrate components out of order and prioritize the one the user clicked.

💡 Memory Hook: Hydration = "wake up server HTML with JavaScript." Server sends static HTML for fast paint + SEO; React attaches interactivity. Mismatches cause warnings and re-renders.

4What is the React Compiler (React Forget)?

The React Compiler is a build-time tool (Babel plugin) that automatically adds memoization. You write plain, readable code; the compiler analyses it and inserts the equivalent of useMemo/useCallback/React.memo where it would actually help — so you don't have to reason about it manually. 

// You write this (no memo):
function ProductList({ items, query }) {
  const filtered = items.filter(i => i.includes(query));
  const onSelect = (id) => navigate(`/p/${id}`);
  return filtered.map(i => <Card key={i.id} onSelect={onSelect} />);
}
// Compiler rewrites filtered & onSelect to be memoized automatically

The goal: eliminate the entire "should I memoize this?" tax. You write simple code and get the performance of hand-tuned memoization. It enforces the rules of React strictly (pure render, no side effects in render) — impure code that happened to work before can break, which is a feature: it surfaces latent bugs.

💡 Memory Hook: React Compiler = automatic optimization. Write simple code; the compiler adds memoization. No more manual useMemo/useCallback — but your render must be pure.

5What are React Server Actions?

Server Actions are functions that run on the server but are invoked from client components, typically as form action handlers. Marked with the "use server" directive, they let you mutate server state (write to a DB) directly from a form — no manual fetch, no API route boilerplate. 

// Server Action
async function createUser(formData) {
  "use server";
  await db.users.create({ name: formData.get("name") });
  revalidatePath("/users");
}

// Client form uses it directly
<form action={createUser}>
  <input name="name" />
  <button>Add</button>
</form>

A subtle win: progressive enhancement. Because the handler is a real form action, the form works even with JavaScript disabled — the browser POSTs natively. They integrate with useFormStatus and useOptimistic for pending state and optimistic UI (next cards).

💡 Memory Hook: Server Actions = form actions that run on the server. No fetch, no API route. "use server" = runs on server. Works without JavaScript (progressive enhancement).

6What is the difference between controlled and uncontrolled forms (deep)?

Beyond the basic distinction (Lesson 3), in real apps the choice has real consequences:

  • Controlled — every keystroke updates React state. You get full control over validation, formatting, conditional UI, and submission. More re-renders and more code, but predictable and powerful.
  • Uncontrolled — the DOM holds values; you read them via refs at submit time. Less code, fewer re-renders, but no live validation/formatting. Good for simple forms or integrating non-React widgets.
// Controlled — live validation is trivial
<input
  value={email}
  onChange={e => setEmail(e.target.value)}
/>
{email && !isEmail(email) && <span>Invalid</span>}

// Uncontrolled — read at submit
const ref = useRef();
<form onSubmit={() => submit(ref.current.value)}>
  <input ref={ref} />
</form>

In practice, libraries like React Hook Form bridge both — uncontrolled by default for performance, with controlled-style validation when you need it. For complex forms with interdependent fields and live validation, controlled is the clearer mental model.

💡 Memory Hook: Controlled for complex forms (validation, formatting, conditional UI). Uncontrolled for simple forms or non-React integrations. React Hook Form blends both.

7What is the useFormStatus hook?

useFormStatus (React 19) gives a component the pending state of its parent <form> — without you threading a loading prop through. It must be rendered inside a <form> (typically as a child component, not the form itself). 

function SubmitButton() {
  const { pending } = useFormStatus();
  return (
    <button disabled={pending}>
      {pending ? "Submitting…" : "Submit"}
    </button>
  );
}

// Usage — no pending prop needed:
<form action={serverAction}>
  <SubmitButton />
</form>

It exposes pending, data, method, and action — enough to build a fully-featured submit button without lifting the form's state. The catch: it reads the parent form, so the component calling it must be a descendant of <form>, not the form itself.

💡 Memory Hook: useFormStatus = "is my parent form submitting?" Must be a child of <form>. No need to pass loading state as props.

8What is the useOptimistic hook?

useOptimistic (React 19) shows an optimistic UI update while an async operation is in flight, and reverts automatically if the operation fails or the real value arrives. It dramatically improves perceived performance — the UI reacts instantly rather than waiting for the server. 

const [optimisticName, addOptimistic] = useOptimistic(
  name,
  (state, newName) => newName // reducer: how to apply the update
);

async function updateName(formData) {
  const newName = formData.get("name");
  addOptimistic(newName);   // instant UI update
  await saveToServer(newName); // real update
}

During the await, the UI shows optimisticName (the new value). Once the action resolves, React reconciles back to the canonical name from props/state. If it throws, the optimistic value is discarded and the previous value shows. Pairs naturally with Server Actions.

💡 Memory Hook: useOptimistic = "assume it works, revert if it doesn't." Like a restaurant showing your order before the kitchen confirms. Auto-reverts on failure.

9What is the use hook (React 19)?

use is a React 19 API that reads a resource during render — currently a Promise or a Context. It's the only hook-like API you can call conditionally or in a loop, because it's designed to read values exactly where you need them. 

function Profile({ promise }) {
  const user = use(promise);  // suspends until resolved
  return <h1>Hi {user.name}</h1>;
}

// With context:
if (showTheme) {
  const theme = use(ThemeContext); // ✅ allowed in a condition
}

When you use(promise), the component suspends — the nearest <Suspense> boundary shows its fallback until the promise resolves. This is a fundamental shift: data fetching becomes part of the render rather than a side effect in useEffect. There's no dependency array because you're reading a specific resource instance, not synchronizing with one.

💡 Memory Hook: use = read async values during render. Only hook callable in conditions/loops. Works with promises (suspends) and context. No deps array — reads the value you hand it.

🧪 Quick Check

1. Which is true of a React Server Component?

✅ Correct! RSCs run server-side, can fetch data with async/await, and never ship their JS to the browser. Client Components opt in with "use client".
❌ RSCs run on the server, can be async, and ship zero JS. They render the output, not the source. They can render Client Components (but not vice-versa, directly).

2. A hydration mismatch warning fires. What most likely caused it?

✅ Correct! Mismatches happen when server and client disagree — typically non-deterministic values (time, random, browser-only APIs) used during render.
❌ Hydration mismatches mean the server HTML and client's first render disagree — usually from non-deterministic render output (Date.now, Math.random, window-only APIs).

3. What makes the use hook different from all other hooks?

✅ Correct! use can be called in conditions/loops because it reads a specific resource. All other hooks must be called unconditionally at the top level.
❌ use is unique in being callable inside conditions and loops. Every other hook must obey the rules-of-hooks (top-level, unconditional).

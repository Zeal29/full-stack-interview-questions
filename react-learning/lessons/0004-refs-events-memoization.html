Refs, Events & Memoization

useRef, synthetic events, and the memoization trio (useMemo, useCallback, React.memo) — and when to skip them

What You'll Learn

  • Use useRef for DOM access and mutable non-state values
  • Explain React's synthetic event system and the camelCase convention
  • Know when useMemo, useCallback, and React.memo actually help
  • Recognize the "over-optimization" anti-pattern

Escaping the Render Cycle

So far everything has been declarative: state changes → React re-renders. But sometimes you need to step outside that loop — to focus an input, measure a node, store a timer id, or hold a mutable value that should not trigger a render. That's useRef. And sometimes a component re-renders more than it needs to, and you want to cache work — that's the memoization trio.

The most important interview insight about memoization: it's not free. useMemo/useCallback cache values but cost a dependency check on every render; React.memo does a shallow prop comparison. Reach for them only when profiling shows a real problem — the react.dev guidance is explicit: "you might not need memo." React 19's compiler will eventually do this automatically (Lesson 9).

1What is the useRef hook?

useRef returns a mutable object ({ current }) that persists across renders without causing re-renders. Two main uses: (1) access a DOM node by passing the ref as the ref prop; (2) store any mutable value (a timer id, a "previous value", a flag) that you don't want to trigger renders. 

const inputRef = useRef(null);
<input ref={inputRef} />
<button onClick={() => inputRef.current.focus()}>Focus</button>

// Storing a mutable value (not state):
const timerRef = useRef(null);
timerRef.current = setInterval(tick, 1000); // no re-render

Mutating .current does not schedule a re-render — that's the whole point. If you need the UI to update when the value changes, that's useState, not useRef.

💡 Memory Hook: useRef = "remember this, but don't re-render on change." Use for DOM access, timers, previous values. NOT for data that should drive the UI.

2What are React events?

React events use synthetic events — a cross-browser wrapper around native events that normalizes behaviour (so you write one API, not IE-vs-everyone-else). Handlers are attached via camelCase props (onClick, onChange, onSubmit), and you pass a function reference, not a function call. 

<button onClick={handleClick}>Click</button>   // ✅ reference
<button onClick={handleClick()}>X</button> // ❌ runs on render

<form onSubmit={e => {
  e.preventDefault(); // stop full-page reload
}}>…</form>

Since React 17, events are attached to the root container (not document), which fixes issues with nested React apps and avoids some event-delegation surprises. Synthetic events are pooled in older React (16) but not anymore — you no longer need e.persist().

💡 Memory Hook: React events = camelCase (onClick), pass the function not a call ({fn} not {fn()}), and use e.preventDefault() instead of return false.

3What is the useMemo hook?

useMemo caches the result of an expensive calculation and only recomputes when its dependencies change. Without it, the calculation would run on every render even if inputs are identical. 

const sortedItems = useMemo(
  () => items.toSorted((a, b) => a.name.localeCompare(b.name)),
  [items] // only recompute when items changes
);

When it helps: genuinely expensive work (sorting thousands of rows, heavy math, deriving big objects) where the cost of computing exceeds the cost of the dependency check + cache lookup. When it hurts: wrapping trivial operations — the memo machinery costs more than the saved work. The react.dev advice: measure first, memo second.

💡 Memory Hook: useMemo = "cache this calculation." Use for genuinely expensive work, not simple ones. Overusing it makes code slower and harder to read.

4What is the useCallback hook?

useCallback caches a function reference so it keeps the same identity across renders (unless its dependencies change). It's the function counterpart to useMemo (which caches a value). Use it when that stable identity matters — specifically, when the function is a dependency of useEffect or a prop to a memoized child. 

const handleClick = useCallback(() => {
  setCount(c => c + 1);
}, []); // stable identity across renders

// Only meaningful because Button is memoized:
<Button onClick={handleClick} /> // same fn → memo skip works

Common misconception: wrapping every function in useCallback "for performance". A function is recreated on each render anyway — caching it only helps if some other code compares it by reference (a memoized child's prop check, or a useEffect dep). On its own, useCallback adds overhead.

💡 Memory Hook: useCallback = "don't recreate this function." Only useful when the function's identity matters — i.e. it's a dependency of useEffect or a prop to a memoized child.

5What is React.memo?

React.memo is a higher-order component that skips re-rendering a child when its props are unchanged. It does a shallow comparison of props by default; pass a custom comparator as the second argument if you need deeper control. 

const Button = React.memo(({ label, onClick }) => (
  <button onClick={onClick}>{label}</button>
));

The trap: if the parent passes a new object/array/function every render, the shallow compare fails and memo never skips. That's why useMemo/useCallback pair with memo — they keep prop identity stable. Reach for memo only after profiling shows wasted renders; don't memoize everything.

💡 Memory Hook: React.memo = "skip re-render if props unchanged." Shallow-compares props. Only use when profiling shows unnecessary re-renders — and pair with stable props (useMemo/useCallback) or it's useless.

🧪 Quick Check

1. Mutating myRef.current = 5 causes…

✅ Correct! Refs hold mutable values that bypass the render cycle. If you need a render, use useState.
❌ Refs never trigger re-renders. If a change should update the UI, use useState, not useRef.

2. React.memo compares props by…

✅ Correct! Shallow === per prop — that's why a parent passing a fresh inline function/object defeats memo, and why useCallback/useMemo pair with it.
❌ React.memo does a shallow per-prop ===. Any new object/array/function reference each render will defeat it.

3. When does useCallback actually help performance?

✅ Correct! The cached identity only matters if something compares it by reference — a useEffect dep or a React.memo child.
❌ Caching a function reference only helps when that identity is compared — as a useEffect dependency or a prop to a React.memo child. Otherwise it's pure overhead.

State & Core Hooks

State, useState, useEffect, and the component lifecycle — the engine that makes React reactive

What You'll Learn

  • Distinguish state from props and explain why state triggers re-renders
  • Use useState correctly, including the functional-updater pattern
  • Use useEffect with the right dependency array and cleanups
  • Map the three lifecycle phases onto hooks

State Makes React Reactive

Props are data from the outside; state is a component's private memory. When state changes, React re-runs the component function and reconciles the new output against the DOM. This is the entire reactivity model: state change → re-render → DOM update. The react.dev docs call this "adding interactivity".

Two hooks do most of the work in any React app: useState (hold data that changes) and useEffect (synchronize with things outside React — the network, the DOM, subscriptions). The single biggest source of React bugs is misunderstanding when effects run and when state updates are applied. Get these two right and most interview traps dissolve.

1What is state in React?

State is mutable data managed within a component. When state changes, React re-renders that component (and its children). Use the useState hook in function components. State is private to the component that owns it and persists across re-renders — React stashes it outside the function call.

Contrast with props (external, read-only data passed in). State is internal and changeable by the component itself; props come from the parent and cannot be mutated. 

const [count, setCount] = useState(0);
// count  = current value (read it in JSX)
// setCount = updater — calling it triggers a re-render
💡 Memory Hook: State = the component's memory. Change it → React re-renders. Props = external input; state = internal memory.

2What is the useState hook?

useState adds local state to a function component. It returns a tuple [currentValue, setterFunction]. The initial value is used only on the first render — subsequent renders ignore it. To update based on the previous value, use the functional updater form so you never close over stale state. 

const [name, setName] = useState("Ali");

// Direct set
setName("Ahmed");

// Functional update — needed when new state depends on previous
setCount(prev => prev + 1);

// Lazy initial value — runs once, avoids expensive work on every render
const [data, setData] = useState(() => expensiveInit());

The functional form matters because state updates are batched and asynchronous. If you call setCount(count + 1) twice in a row, both read the same count, so you get +1 not +2. With setCount(prev => prev + 1) each call sees the latest queued value.

💡 Memory Hook: Always use the callback form (setCount(prev => prev + 1)) when the new state depends on the previous. It prevents stale-closure bugs under batching.

3What is the useEffect hook?

useEffect runs side effects after the component renders — fetching data, subscribing, manually touching the DOM, setting timers. It takes a function and an optional dependency array. You can return a cleanup function that runs before the next effect and on unmount. 

useEffect(() => {
  const id = setInterval(tick, 1000);
  return () => clearInterval(id); // cleanup on unmount / before re-run
}, []); // dependency array

The dependency array controls when the effect re-runs:

  • [] — run once after mount (and cleanup on unmount).
  • [a, b] — run after mount and whenever a or b changes.
  • no array — run after every render (almost never what you want).

Think of effects as "synchronize this component with some external system", not as a lifecycle lifecycle hook. The react.dev guidance: don't chain effects together — derive what you can during render.

💡 Memory Hook: useEffect = "do something after render." [] = mount only. [x] = when x changes. No array = every render (rarely wanted). Always clean up subscriptions/timers — a missing cleanup is the #1 cause of memory leaks and stale-state bugs.

4What is the component lifecycle in React?

Three phases:

  1. Mounting — component added to the DOM. Hook equivalent: useEffect(fn, []).
  2. Updating — state or props change, causing a re-render. Hook equivalent: useEffect(fn, [deps]).
  3. Unmounting — component removed from the DOM. Hook equivalent: the cleanup returned from an effect.

Class components exposed explicit methods: componentDidMount, componentDidUpdate, componentWillUnmount. Hooks unify all three into effects with the dependency array. (There's also a "render phase" vs "commit phase" distinction — render produces the virtual DOM, commit applies it to the real DOM; effects run after commit.) 

// All three lifecycle phases in one hook:
useEffect(() => {
  console.log("mount OR dep changed"); // mount + update
  return () => console.log("cleanup");        // unmount + before re-run
}, [dep]);
💡 Memory Hook: In hooks: Mount = useEffect(fn, []). Update = useEffect(fn, [deps]). Unmount = return () => cleanup inside the effect. Three class methods collapse into one hook.

🧪 Quick Check

1. You call setCount(count + 1) twice in the same handler. The final value of count is…

✅ Correct! Both calls close over the same count, so you get +1. Use the functional form setCount(prev => prev + 1) to get +2.
❌ Both calls read the same count from this render (updates are batched/async). Result is +1. Use setCount(prev => prev + 1) instead.

2. When does a useEffect with an empty dependency array [] run?

✅ Correct! [] means "no dependencies to watch" → run once after mount, cleanup on unmount.
❌ [] means "watch nothing" → the effect runs once after mount and its cleanup runs once on unmount.

3. What happens if you omit the dependency array entirely?

✅ Correct! With no array, the effect runs after every render — almost always a bug (often an infinite loop if the effect sets state).
❌ No array = run after every render. If the effect sets state, this creates an infinite re-render loop — a classic interview bug.

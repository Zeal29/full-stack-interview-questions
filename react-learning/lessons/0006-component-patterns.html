Component Patterns

HOCs, render props, error boundaries, portals, and compound components — the patterns interviewers love

What You'll Learn

  • Explain HOCs and render props and why hooks largely replaced them
  • Implement an error boundary (and know it must be a class)
  • Use portals to render outside the parent DOM tree
  • Build compound components that share state implicitly via context

Patterns for Sharing Behavior

Before hooks existed, sharing stateful logic across components required Higher-Order Components (wrapping a component in a function) or render props (passing a function that returns JSX). Hooks replaced most of these use cases — a custom hook (Lesson 7) is almost always simpler. But you still need to recognize HOCs and render props in older codebases, and a few composition problems are still best solved with them.

Error boundaries and portals solve different problems: boundaries catch render-time crashes (and only class components can be one), portals render DOM into a different container (modals, tooltips). Compound components give you the elegant "<Tabs><Tabs.List><Tabs.Tab/></Tabs.List>" API you see in design systems.

1What are Higher-Order Components (HOC)?

An HOC is a function that takes a component and returns an enhanced component. It's a pattern for cross-cutting concerns — auth gating, logging, data fetching — wrapping a component with extra behaviour without modifying it. 

const withAuth = (Component) => {
  return (props) => {
    if (!isLoggedIn()) return <Login />;
    return <Component {...props} />;
  };
};
const ProtectedPage = withAuth(Dashboard);

Status: less common since hooks — a custom useAuth() hook is usually cleaner. Still useful for cross-cutting concerns that genuinely wrap the whole component (e.g. injecting a provider). Famous example: connect() from react-redux is an HOC. Watch for "wrapper hell" (deeply nested HOCs) and lost displayName — the react.dev guidance is to prefer hooks for new code.

💡 Memory Hook: HOC = wrapper component — takes a component, returns a better one. Hooks replaced most HOC use cases; still useful for complex cross-cutting wrapping.

2What are render props?

A render prop is a prop whose value is a function that returns JSX. The parent component owns some state/logic and calls the function to let the consumer decide how to render it. The most common form is the children prop used as a function. 

<DataProvider render={(data) => <List items={data} />} />

// Or via children:
<DataProvider>
  {(data) => <List items={data} />}
</DataProvider>

Like HOCs, render props were the pre-hooks way to share stateful logic. A custom hook does the same job without the nesting. You'll still see render props in older libraries (react-router v5's <Route render={...} />) and in some animation libraries, so recognising the pattern matters.

💡 Memory Hook: Render props = "you tell me how to render, I give you the data." Hooks replaced this pattern, but it lingers in older codebases.

3What is a React error boundary?

An error boundary is a component that catches JavaScript errors thrown during rendering, in lifecycle methods, and in constructors of child components, then displays a fallback UI instead of unmounting the whole tree. Only class components can be error boundaries — there is no hook equivalent (yet). 

class ErrorBoundary extends React.Component {
  state = { hasError: false };

  static getDerivedStateFromError(error) {
    return { hasError: true };
  }

  componentDidCatch(error, info) {
    logError(error, info); // side-effect: telemetry
  }

  render() {
    return this.state.hasError
      ? <Fallback />
      : this.props.children;
  }
}

What they DON'T catch: errors in event handlers, async code (setTimeout, promises), or errors thrown in server-side rendering. For those, use try/catch directly. The community library react-error-boundary wraps the boilerplate.

💡 Memory Hook: Error boundary = try/catch for JSX. Must be a class component. Catches render/lifecycle errors only — NOT event handlers, async, or SSR errors.

4What are React portals?

A portal renders children into a DOM node outside the parent component's DOM hierarchy, while keeping them inside the React component tree. Useful for modals, tooltips, and overlays that visually escape their container (and would otherwise be clipped by overflow: hidden or broken by z-index stacking). 

ReactDOM.createPortal(
  <Modal />,
  document.getElementById("modal-root")
);

The key win: the child stays in the React tree, so events still bubble through the React hierarchy (not just the DOM hierarchy) and context still works — even though the actual DOM nodes are attached elsewhere. That lets you build a modal that's a true child in React but renders at document.body.

💡 Memory Hook: Portal = "render here, show there." Modal stays in the React tree (events bubble, context works) but appears in a different DOM location.

5What is the compound components pattern?

Compound components are a set of components that implicitly share state through context, exposing a clean, composable API. Think HTML's <select> + <option>: the options know about the select's state without explicit props. This is how design systems like Radix, Headless UI, and Mantine build Tabs, Accordion, Menu, and Dialog. 

<Tabs>
  <Tabs.List>
    <Tabs.Tab>Tab 1</Tabs.Tab>
    <Tabs.Tab>Tab 2</Tabs.Tab>
  </Tabs.List>
  <Tabs.Panels>
    <Tabs.Panel>Content 1</Tabs.Panel>
    <Tabs.Panel>Content 2</Tabs.Panel>
  </Tabs.Panels>
</Tabs>

The parent (Tabs) owns the active index in state and provides it via context; each sub-component consumes that context. The result: flexible composition (consumers arrange children however they want) and an API that reads like the final HTML. The trade-off is more setup code than a single prop-driven component.

💡 Memory Hook: Compound components = implicit state sharing. Like <select> + <option>. Children know about parent state through context, without explicit props.

🧪 Quick Check

1. Which errors does an error boundary NOT catch?

✅ Correct! Boundaries catch render/lifecycle/constructor errors only. Event handlers, async code, and SSR errors slip through — use try/catch there.
❌ Boundaries catch render-time and lifecycle errors. They do NOT catch errors in event handlers, setTimeout, or promise rejections — wrap those in try/catch.

2. You render a Modal via a portal to document.body. What stays true?

✅ Correct! A portal only changes the DOM mount point. The component stays in the React tree, so event bubbling and context behave as if it were still nested.
❌ A portal moves only the DOM node. React keeps the component in the tree, so events bubble through React's hierarchy and context resolves normally.

3. Compound components share state primarily through…

✅ Correct! The parent owns state and exposes it via context; each sub-component reads what it needs. That's what makes the API read like native HTML.
❌ Compound components use Context — the parent provides state, the sub-components consume it. No HOC, no global store, no drilling.

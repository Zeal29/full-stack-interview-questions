Handling Input and Lists

Two questions come up in nearly every React interview: "where does the form data live?" and "why did my list break when I deleted an item?". The first is the controlled-vs-uncontrolled choice; the second is almost always a key problem. Both reduce to a single principle: React needs a stable identity for data so it can match old UI to new UI.

For inputs, "controlled" means React state is the single source of truth — every keystroke flows through state and back into the input's value. For lists, a stable key tells the reconciler "this row is the same row as before, just moved/updated" — without it, React reuses DOM nodes by position and you get the infamous "wrong row edited" bug.

1What is the difference between controlled and uncontrolled components?

Controlled: React state drives the input's value, and an onChange updates that state. React is the single source of truth — the input can't change without state changing.

Uncontrolled: the DOM owns the value; React reads it on demand via a ref. You set a default with defaultValue and pull inputRef.current.value at submit time. 

// Controlled — React state is the source of truth
<input value={name} onChange={e => setName(e.target.value)} />

// Uncontrolled — DOM is the source, read via ref
const inputRef = useRef();
<input defaultValue="" ref={inputRef} />
// read later: inputRef.current.value
💡 Memory Hook: Controlled = React drives (state = value). Uncontrolled = DOM drives (ref to get value). Prefer controlled for most forms — it gives you validation, formatting, and predictable submission.

2What are keys in React lists, and why does the choice of key matter?

Keys are stable identities React uses to match list items across renders. When a list re-renders, the reconciler uses keys to decide whether an existing DOM node can be reused (updated in place) or must be recreated. Use a unique, stable id from your data — never the array index. 

// ✅ stable unique id
items.map(item => <li key={item.id}>{item.name}</li>);

// ❌ index — breaks on reorder/insert/delete
items.map((item, i) => <li key={i}>{item.name}</li>);

With index keys, deleting the first item makes React think "item at index 0 changed content" rather than "item 0 was removed" — so it updates DOM nodes in place, transferring state (e.g. a checkbox tick, or uncontrolled input text) to the wrong row. The bug is subtle and the fix is always a stable key.

💡 Memory Hook: Keys = name tags for list items. Use a unique id from the data; never the index. Wrong keys = state leaking between rows, weird re-render bugs, and lost performance.

3What is conditional rendering in React?

Conditional rendering means showing different UI based on a condition, expressed inline in JSX. Three idioms cover nearly every case: 

// 1. Ternary — either A or B
{isLoggedIn ? <Dashboard /> : <Login />}

// 2. && — show or hide (note: avoid 0/"" leaking in)
{items.length > 0 && <ItemList />}

// 3. Extract complex logic to a helper / early return
function Greeting({ user }) {
  if (!user) return <Login />;
  return <h1>Hi {user.name}</h1>;
}

Gotcha: {count && <Foo/>} renders 0 when count is 0, because 0 is a falsy but renderable value. Guard with {count > 0 && ...} or coerce to boolean.

💡 Memory Hook: Ternary (? :) = either A or B. && = show or hide. Don't put if inside JSX — use it above the return or extract a helper.

4What is a React Fragment?

A Fragment (<Fragment> or the shorthand <>) groups multiple elements without adding an extra DOM node. JSX requires a single root element per return; a Fragment satisfies that rule without polluting the DOM with a wrapper div that could break CSS (flex/grid) or valid HTML (e.g. <tr> can only contain <td>, not a wrapping div). 

// Shorthand — no key support
return (
  <>
    <td>Cell 1</td>
    <td>Cell 2</td>
  </>
);

// Explicit — needed when mapping (key goes on Fragment)
items.map(item => (
  <React.Fragment key={item.id}>
    <dt>{item.term}</dt>
    <dd>{item.def}</dd>
  </React.Fragment>
));
💡 Memory Hook: Fragment = invisible wrapper. Use <> when a real parent would break layout or produce invalid HTML (like a div between table and tr). Use the explicit form with key when mapping.

🧪 Quick Check

1. In a controlled input, what is the single source of truth for the input's value?

✅ Correct! Controlled = React state is the source; the value prop mirrors state and onChange updates it.
❌ Controlled means React state owns the value. The DOM is just rendering what state says.

2. You use the array index as key and delete the middle item of a list. What can go wrong?

✅ Correct! Index keys mean "node at position N" — deleting shifts everything, so React updates nodes in place and per-item state lands on the wrong row.
❌ Index keys confuse the reconciler after reorder/delete: DOM nodes get reused by position, so per-row state (input text, ticks) drifts to the wrong item. Use a stable id.

3. Why use a Fragment instead of a wrapping <div>?

✅ Correct! Fragments group children without a DOM node — critical for valid table markup and for not breaking flex/grid parents.
❌ Fragments exist to avoid extra DOM nodes — a wrapper div can break flex/grid layout or produce invalid HTML (e.g. div inside tbody).

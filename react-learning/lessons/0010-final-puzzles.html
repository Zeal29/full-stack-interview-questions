🟢 Easy Puzzles

Puzzle 1 — Predict the Output easy

A button click handler calls setCount three times in a row. After the click, what does count render as — and why? 

function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    setCount(count + 1);
    setCount(count + 1);
    setCount(count + 1);
  }

  return <button onClick={handleClick}>{count}</button>;
}

The button shows 1, not 3. All three calls read the same count from this render (which is 0), so each computes 0 + 1. React batches them into a single re-render with the final value 1.

The fix — use the functional updater form:

function handleClick() {
  setCount(c => c + 1); // sees 0 → queues 1
  setCount(c => c + 1); // sees 1 → queues 2
  setCount(c => c + 1); // sees 2 → queues 3 → renders 3
}

Why: state updates are batched and asynchronous. During a single event handler, count is a snapshot from the current render and doesn't change. The functional updater receives the latest queued value, so each call builds on the previous one. This is the #1 React gotcha and an extremely common interview question.

💡 Key Insight: State variables are snapshots per render. When the new state depends on the old, always use the functional form setX(prev => ...) — it's the only way to safely chain updates under batching.
Puzzle 2 — Spot the Infinite Loop easy

This component crashes the browser tab with an infinite re-render loop. What's the bug, and what's the minimal fix? 

function Profile({ id }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    fetchUser(id).then(setUser);
  }); // ← no dependency array

  return <div>{user?.name}</div>;
}

The effect has no dependency array, so it runs after every render. The flow: render → effect runs → setUser → re-render → effect runs again → setUser → … forever. Each fetch resolves and triggers another render, which triggers another fetch.

Minimal fix — add the dependency array:

useEffect(() => {
  fetchUser(id).then(setUser);
}, [id]); // only re-run when id changes

Why: the dependency array is the effect's contract for "when should this re-run?" [id] means "run on mount, and again only if id changes." Without it, React assumes the effect depends on everything and re-runs it every render. Even better: add an ignore flag to avoid setting state after unmount, and handle the loading/error states. 

useEffect(() => {
  let ignore = false;
  fetchUser(id).then(u => { if (!ignore) setUser(u); });
  return () => { ignore = true; };
}, [id]);
💡 Key Insight: The three dependency-array modes: [] = mount only; [deps] = mount + when deps change; no array = every render (almost always a bug). Setting state in a no-array effect is the textbook infinite-loop bug.
Puzzle 3 — The Disappearing Checkbox easy

A list of todos uses the array index as key. Each row has a checkbox. You check the first row's box, then delete the first todo. Describe the visual bug and explain why it happens. 

function TodoList({ todos, onRemove }) {
  return (
    <ul>
      {todos.map((todo, index) => (
        <li key={index}>
          <input type="checkbox" /> {/* uncontrolled */}
          {todo.text}
          <button onClick={() => onRemove(index)}>x</button>
        </li>
      ))}
    </ul>
  );
}

The bug: the checked state appears to "move down" to the new first row. After deletion, the second todo's row shows the checkbox as checked, even though you never touched it.

Why: with index keys, React identifies rows by position, not identity. Deleting the first item shifts everything up by one. React sees "the row at index 0 still exists" and reuses its DOM node — including the uncontrolled checkbox's internal checked state — then updates only the text to the new todo. The checkbox DOM element is reused, so its checked state persists on the wrong row.

Fix — use a stable unique id from the data:

{todos.map(todo => (
  <li key={todo.id}>
    <input type="checkbox" />
    {todo.text}
  </li>
))}

With stable keys, React sees "the row with id: 5 was removed" and unmounts its DOM node entirely (including its checkbox state) — instead of shuffling state between reused nodes.

💡 Key Insight: Keys are about identity, not ordering. Index keys conflate "position" with "identity," so any reorder/insert/delete transfers per-row state (checkboxes, uncontrolled inputs, animations) to the wrong item. Always use a stable id from the data.
Puzzle 4 — Props or State? easy

You're building a <Counter initialCount={5} /> component. The parent passes initialCount, and the user can click +/− to change the displayed number. Which data should be props, which should be state, and why?

initialCount is a prop (the parent owns it; the child only reads it once). The current count is state — it's internal data the component owns and mutates, and it's what drives re-renders. 

function Counter({ initialCount }) {
  const [count, setCount] = useState(initialCount); // seed once
  return (
    <button onClick={() => setCount(c => c + 1)}>
      {count}
    </button>
  );
}

The decision rule:

  • Is the value passed in by the parent? → props.
  • Does the value change over time, driven from inside this component? → state.
  • Can you compute it from existing state/props? → neither (derive it during render).

Gotcha: if initialCount changes later, the counter will not reset, because useState(initialCount) only reads the initial value once. If you need it to reset when the prop changes, either use a key to remount the component, or sync with useEffect (the "key to reset" approach is preferred and cleaner).

💡 Key Insight: Props = external input (read-only). State = internal memory (mutable by the component). The initial* pattern seeds state from a prop once; to re-seed, change the component's key rather than fighting useState.
Puzzle 5 — Effect Timing easy

What does this component log to the console on its first render, and in what order? 

function App() {
  console.log("A: during render");

  useEffect(() => {
    console.log("B: in effect");
  }, []);

  return <div>hi</div;
}

Order: "A: during render" first, then "B: in effect" — after the browser has painted.

Why: React's render cycle is render → commit → paint → effect. The component function runs ("A"), React commits the virtual DOM to the real DOM, the browser paints the screen, and only then do passive effects (useEffect) fire ("B"). So effects run after paint, asynchronously — they never block the user from seeing the initial UI.

This is why useEffect is the right default for data fetching: the user sees the initial render (often a skeleton) immediately, and the fetch kicks off after paint. If you needed to read DOM layout before paint to avoid flicker, you'd reach for useLayoutEffect instead — it runs synchronously between commit and paint.

💡 Key Insight: The lifecycle per render: render → commit (DOM mutated) → browser paint → passive effects (useEffect). Effects run after paint; that's why they're non-blocking and why reading layout in them can cause a one-frame flicker.

🟡 Medium Puzzles

Puzzle 6 — The Stale Counter medium

This timer is supposed to log the current count every second — but it always logs 0, no matter how many times you click. Diagnose the bug and give two valid fixes. 

function Timer() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      console.log(count); // always logs 0
    }, 1000);
    return () => clearInterval(id);
  }, []); // ← empty deps

  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}

The bug — a stale closure. The effect runs once (empty deps) and captures count from that first render, which is 0. The interval's callback closes over that original count forever. Later renders create new count values, but the interval is still referencing the old one.

Fix 1 — add count to the dependency array. The effect re-runs (and resets the interval) whenever count changes:

useEffect(() => {
  const id = setInterval(() => console.log(count), 1000);
  return () => clearInterval(id);
}, [count]); // ← fresh closure each time count changes

Fix 2 — use a ref to hold the latest value (avoids resetting the interval):

const countRef = useRef(count);
useEffect(() => { countRef.current = count; }); // keep ref fresh every render

useEffect(() => {
  const id = setInterval(() => console.log(countRef.current), 1000);
  return () => clearInterval(id);
}, []); // interval set up once; reads latest via ref

Fix 3 (if updating state) — functional updater: if the interval was doing setCount(count + 1), switch to setCount(c => c + 1) — no closure on count at all. Choose Fix 1 when re-running is cheap; Fix 2 when the setup (WebSocket, expensive subscription) shouldn't be torn down.

💡 Key Insight: A "stale closure" is when a long-lived function (interval, subscription, event listener) captures variables from the render it was created in, never seeing updates. Three fixes: re-run the effect (add deps), read via a ref, or use the functional updater form for state.
Puzzle 7 — Design useDebounce medium

Design a custom hook useDebounce(value, delay) that returns a debounced copy of value — it only updates after value has stopped changing for delay ms. Write the implementation and explain each line. 

// Target API:
const debouncedQuery = useDebounce(query, 300);
function useDebounce(value, delay) {
  const [debounced, setDebounced] = useState(value);

  useEffect(() => {
    // 1. Schedule the update after `delay` ms
    const id = setTimeout(() => {
      setDebounced(value);
    }, delay);

    // 2. Cleanup: clear the timer if value changes again before delay elapses
    return () => clearTimeout(id);
  }, [value, delay]); // re-run when value OR delay changes

  return debounced;
}

How it works, line by line:

  1. useState(value) — hold the debounced value, seeded with the initial value.
  2. The effect runs whenever value changes (it's a dependency).
  3. setTimeout schedules the actual update delay ms later.
  4. The cleanup clears the pending timer. So if value changes again before delay elapses, the effect re-runs, the cleanup cancels the previous timer, and a new one starts — only the last change survives the full delay.

Classic use case: a search input that triggers an API call. You debounce the query so you don't fire a request on every keystroke — only after the user pauses typing.

💡 Key Insight: Debounce via useEffect + setTimeout + cleanup is a canonical custom-hook pattern. The cleanup is what makes it debounce — each new change cancels the previous pending update, so only the final value in a burst gets through.
Puzzle 8 — Context Re-render Storm medium

This app uses a single AppContext holding { user, theme, cartCount }. Whenever cartCount changes, every consumer re-renders — even components that only read theme. Why, and how do you fix it? Give at least two strategies. 

const AppContext = createContext();

<AppContext.Provider value={{ user, theme, cartCount }}>
  <App />
</AppContext.Provider>

Why: the provider passes a brand-new object literal on every render. Consumers compare the context value by reference — and {user, theme, cartCount} is a new object each time, so every consumer re-renders whenever the provider itself re-renders, even if the individual fields didn't change.

Fix 1 — memoize the value object:

const value = useMemo(
  () => ({ user, theme, cartCount }),
  [user, theme, cartCount]
);
<AppContext.Provider value={value}>

Now the value keeps the same reference unless one of those fields actually changes.

Fix 2 — split contexts by update frequency:

<UserContext.Provider value={user}>        // rarely changes
  <ThemeContext.Provider value={theme}>    // rarely changes
    <CartContext.Provider value={cartCount}>  // changes often
      <App />
    </CartContext.Provider>
  </ThemeContext.Provider>
</UserContext.Provider>

A component reading only useContext(ThemeContext) won't re-render when the cart changes.

Fix 3 — use an external store (Zustand, Jotai, Redux) with useSyncExternalStore. These let a component subscribe to a slice of state and re-render only when that slice changes — the cleanest fix for high-frequency updates.

💡 Key Insight: Context compares its value by reference. A new object literal every render defeats memoization and re-renders all consumers. For low-frequency data: memoize or split contexts. For high-frequency data: use an external store with slice subscriptions.
Puzzle 9 — The Over-Memoized Component medium

A developer wraps a trivial addition in useMemo, "for performance". Does this help or hurt? Explain. 

function Total({ a, b }) {
  const sum = useMemo(() => a + b, [a, b]);
  return <div>{sum}</div>;
}

It hurts. The calculation a + b is essentially free — far cheaper than the memo machinery React runs on every render (creating the dependency array, comparing a and b by reference, and checking the cache). You've added overhead to "save" work that was never expensive.

Rule of thumb: useMemo earns its keep only when the computation is genuinely costly — sorting/filtering large arrays, heavy math, building large objects, or when the memoized value is consumed as a dependency by useEffect or passed as a prop to a memoized child (where identity matters).

The original, faster version:

function Total({ a, b }) {
  const sum = a + b; // just compute it
  return <div>{sum}</div>;
}

The react.dev guidance is explicit: "You might not need an Effect / you might not need memo." Reach for memoization after profiling shows a real bottleneck, not preemptively. The React Compiler (Lesson 9) is designed to remove this tax entirely by memoizing automatically where it helps.

💡 Key Insight: Memoization isn't free — it trades computation for cache + comparison overhead. Only memoize when the saved work exceeds the memo's own cost, or when identity stability is required (deps / memoized children). Measure first.
Puzzle 10 — Does the Boundary Catch It? medium

A component is wrapped in an error boundary. The user clicks a button whose handler throws. Does the boundary catch the error and show the fallback? Explain. 

<ErrorBoundary>
  <form onSubmit={(e) => {
    e.preventDefault();
    saveToServer(data); // ← this throws
  }}>
    <button>Save</button>
  </form>
</ErrorBoundary>

No. The boundary does not catch the error. The error propagates as an uncaught exception (and the app likely crashes or the global window.onerror fires). The fallback UI never shows.

Why: error boundaries catch errors thrown during rendering, in lifecycle methods, and in constructors of child components. They explicitly do NOT catch:

  • Errors in event handlers (like onSubmit, onClick)
  • Errors in async code (setTimeout, await, promise rejections)
  • Errors in server-side rendering

Fix — use try/catch in the handler:

<form onSubmit={async (e) => {
  e.preventDefault();
  try {
    await saveToServer(data);
  } catch (err) {
    setSubmitError(err); // show inline error UI
  }
}}>

Event-handler errors are a regular JS exception, not a React render error — handle them with plain try/catch and update state to show feedback. This is one of the most common misconceptions in React interviews.

💡 Key Insight: Error boundaries are try/catch for the render phase only — not for event handlers, async code, or SSR. The mental model: if it happens during React's render/commit, the boundary catches it; if it happens in response to a user action or in a callback, use try/catch.

🔴 Hard Puzzles

Puzzle 11 — Diagnose the Re-render Storm hard

A <Dashboard> holds query in state. Every keystroke re-renders <Sidebar>, <Header>, and a heavy <ResultsTable> — even though only the table cares about query. Identify all the causes and prescribe fixes. 

function Dashboard() {
  const [query, setQuery] = useState("");
  return (
    <>
      <Header onSearch={setQuery} />
      <Sidebar />
      <ResultsTable query={query} filter={(item) => item.includes(query)} />
    </>
  );
}

Causes (there are several):

  1. Default behavior: when Dashboard re-renders, React re-renders all children by default — including <Header> and <Sidebar> that don't use query.
  2. New filter function each render: the inline arrow (item) => item.includes(query) is recreated every render — even if ResultsTable were memoized, this new reference would defeat the memo.
  3. State location: query lives high in the tree, so its updates ripple down everything below.

Fixes, in order of impact:

// 1. Memoize children that don't depend on query:
const Sidebar = React.memo(function Sidebar() { ... });
const Header = React.memo(function Header({ onSearch }) { ... });

// 2. Stabilize onSearch so Header's memo actually skips:
const handleSearch = useCallback((q) => setQuery(q), []);

// 3. Memoize ResultsTable AND give it a stable filter:
const ResultsTable = React.memo(function ResultsTable({ query }) {
  const rows = useMemo(
    () => data.filter(item => item.includes(query)),
    [data, query]
  );
  // ... render rows
});

Bigger-picture fixes: move the query state down into a <SearchSection> that owns both the input and the table, so Sidebar/Header never see it. Or use useDeferredValue/useTransition (Lesson 8) to make the table update non-urgent. The deepest lesson: profile first — the React DevTools Profiler shows you exactly which components waste renders and why.

💡 Key Insight: Re-render storms usually have three intertwined causes: default child re-rendering, unstable prop references (inline functions/objects), and state held too high. The fix stack: React.memo + stable props (useCallback/useMemo) + state colocation + concurrent APIs for the expensive parts.
Puzzle 12 — Design a Form Architecture hard

You're building a multi-section form with 20+ fields, cross-field validation (e.g. "endDate after startDate"), dynamic fields (add/remove rows), and async submit. How do you structure the state — useState vs useReducer? Controlled vs uncontrolled? Walk through the decision.

State container: useReducer for the form values.

With 20+ interdependent fields, separate useState calls become unmanageable, and cross-field validation is awkward. A single reducer holding { values, errors, touched, isSubmitting } keeps related state together and makes transitions explicit and testable: 

const initial = {
  values: { ... },
  errors: {},
  touched: {},
  isSubmitting: false,
};

function formReducer(state, action) {
  switch (action.type) {
    case "SET_FIELD":
      return { ...state, values: { ...state.values, [action.field]: action.value } };
    case "SET_ERRORS":
      return { ...state, errors: action.errors };
    case "ADD_ROW":   // dynamic fields
    case "REMOVE_ROW":
    case "SUBMIT_START":
    case "SUBMIT_DONE":
    default: return state;
  }
}

Controlled vs uncontrolled — split by need:

  • Controlled for fields needing live validation, formatting (phone, currency), or conditional UI (show/hide based on another field). Most fields.
  • Uncontrolled for large free-text areas where every-keystroke re-renders would be costly, or when integrating a non-React widget (a date picker from jQuery). Read on submit.

Validation strategy: validate on blur (per field) and on submit (whole form). Don't validate on every keystroke — it's annoying and slow. Keep the validator pure and outside the reducer so it's testable.

Pragmatic note: for production, reach for React Hook Form + Zod. RHF uses uncontrolled inputs by default (fewer re-renders) with controlled-style validation, and Zod gives you a single schema that drives both runtime validation and TypeScript types. The architecture above is what those libraries implement for you.

💡 Key Insight: Complex forms → useReducer (related state + explicit transitions + testable logic). Mix controlled (live validation) and uncontrolled (heavy/legacy) inputs. Validate on blur + submit, not every keystroke. In production, React Hook Form + Zod encode these decisions for you.
Puzzle 13 — Responsive Search with Transitions hard

A search box filters 10,000 items. Without optimization, every keystroke blocks the main thread and typing feels laggy. Implement it with both useTransition and useDeferredValue, and explain when you'd pick each.

Approach A — useTransition (you control the update site):

function Search() {
  const [query, setQuery] = useState("");
  const [results, setResults] = useState(allItems);
  const [isPending, startTransition] = useTransition();

  function onChange(e) {
    const q = e.target.value;
    setQuery(q);  // urgent — input updates instantly
    startTransition(() => {
      setResults(heavyFilter(allItems, q)); // non-urgent
    });
  }

  return (
    <>
      <input value={query} onChange={onChange} />
      {isPending && <span>Filtering…</span>}
      <List items={results} />
    </>
  );
}

Approach B — useDeferredValue (the value comes from elsewhere, e.g. props):

function Results({ query }) {  // query is a prop
  const deferredQuery = useDeferredValue(query);
  const results = useMemo(
    () => heavyFilter(allItems, deferredQuery),
    [deferredQuery]
  );
  return <List items={results} />;
}

How they keep typing responsive: the input's query update is urgent and renders immediately. The expensive list update is marked low-priority; React can interrupt it to handle the next keystroke, discarding stale renders. The user sees smooth typing; the list catches up between keystrokes.

When to pick which:

  • useTransition when you control where the state is set — you wrap the setResults call and get isPending for a loading indicator.
  • useDeferredValue when the value arrives from outside (props, a library you don't control) and you can't wrap the setter. It's the value-level equivalent.

Both rely on the same Fiber scheduler under the hood — they're priorities, not debounce timers.

💡 Key Insight: useTransition (update-level) and useDeferredValue (value-level) are two faces of concurrent rendering: mark expensive work low-priority so urgent input stays responsive. Pick by whether you control the setter or just receive the value.
Puzzle 14 — Suspense + Error Boundary + Streaming hard

A page has a header (instant), a sidebar (instant), and three independent data-driven panels that each fetch on mount. Design the Suspense and Error Boundary structure so that: panels load independently, one failing panel doesn't blank the page, and the header/sidebar show immediately.

Key principles: (1) wrap each independent async boundary in its own Suspense + Error Boundary so they don't block each other; (2) keep the always-present UI (header, sidebar) outside any Suspense; (3) nest error boundary inside Suspense so the fallback replaces just that panel. 

<Layout>
  <Header />      // instant, outside any Suspense

  <ErrorBoundary fallback={<PanelError />}>
    <Suspense fallback={<PanelSkeleton />}>
      <ProfilePanel />   // suspends while fetching
    </Suspense>
  </ErrorBoundary>

  <ErrorBoundary fallback={<PanelError />}>
    <Suspense fallback={<PanelSkeleton />}>
      <ActivityPanel />
    </Suspense>
  </ErrorBoundary>

  <ErrorBoundary fallback={<PanelError />}>
    <Suspense fallback={<PanelSkeleton />}>
      <StatsPanel />
    </Suspense>
  </ErrorBoundary>
</Layout>

Why this structure works:

  • Independent loading: each <Suspense> isolates its panel — Profile can render while Activity is still loading, without waiting.
  • Independent failure: if StatsPanel's fetch rejects, only that panel's ErrorBoundary catches it and shows <PanelError>; the rest of the page keeps working.
  • Streaming (SSR): with React's streaming server renderer, the server flushes the header/sidebar HTML immediately, then streams each panel's HTML as its data resolves — the user sees content progressively rather than staring at a blank page.

Nesting order matters: Error Boundary outside Suspense means an error in the fallback could escape upward; the conventional order is <ErrorBoundary><Suspense><Component/></Suspense></ErrorBoundary> so the boundary catches errors from both the component and its fallback.

💡 Key Insight: Compose one Error Boundary + one Suspense per independent async unit. Isolation is the goal: each panel loads and fails on its own, and the shell renders immediately. On the server, this enables streaming — flush the shell, stream panels as they resolve.
Puzzle 15 — The use(promise) Deep Dive hard

This React 19 component reads a promise with use. Three questions: (a) How is this different from fetching in useEffect + useState? (b) Why is there no dependency array? (c) How does it interact with Suspense? 

function Profile({ userPromise }) {
  const user = use(userPromise);
  return <h1>Hello, {user.name}</h1>;
}

// Parent creates the promise and passes it in:
<Suspense fallback={<Spinner />}>
  <Profile userPromise={fetchUser(id)} />
</Suspense>

(a) Different from useEffect + useState: the classic pattern is imperative and side-effect-based — you fetch in an effect, store the result in state, and manage loading/error flags yourself:

// Classic: effect + state
function Profile({ id }) {
  const [user, setUser] = useState(null);
  useEffect(() => {
    fetchUser(id).then(setUser);
  }, [id]);
  if (!user) return <Spinner />;
  return <h1>{user.name}</h1>;
}

With use, fetching is part of the render. The component reads the value synchronously (from React's perspective) and React handles the loading state via Suspense. No manual loading state, no effect, no race-condition cleanup. The data dependency is expressed in the component tree, not bolted on.

(b) No dependency array: use isn't synchronizing with anything — it's reading a specific resource instance you hand it. Each render gets the promise as a prop (or creates a memoized one); use just resolves that exact promise. There's nothing to "re-run when X changes" because the caller controls when a new promise arrives. (Contrast with useEffect, which subscribes to a system and must declare what triggers resubscription.)

(c) Suspense interaction: when use(promise) is called on an unresolved promise, the component suspends — React throws a special promise that the nearest <Suspense> boundary catches, showing its fallback. When the promise resolves, React re-renders the suspended subtree with the value. Errors in the promise are caught by the nearest Error Boundary. This is the same mechanism React.lazy uses — promises as first-class render-time values.

The bigger picture: use is the client-side hook into React's new data model, where Server Components (or a client cache) hand promises to components and the component tree declaratively expresses data dependencies. It's a fundamental shift from "fetch in effects" to "read resources during render."

💡 Key Insight: use(promise) makes data fetching part of the render rather than a side effect. No deps array because it reads a specific instance, not a subscription. It suspends the component until resolution, with Suspense + Error Boundary handling loading and failure declaratively. This is the React 19 data model.
